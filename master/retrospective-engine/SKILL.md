# 회고 엔진 (retrospective-engine) v1.0

> "검증 없는 실행은 도박이고, 회고 없는 검증은 낭비다."
> — MGTG 7분류 체계 원칙

> "포스트모텀 문화에 대한 지속적 투자 덕분에 우리는 더 적은 장애를 겪는다."
> — Google SRE Book

---

## 삼형제 스킬 관계도

```
planning-generator (기획)      problem-solver (실행)      retrospective-engine (회고)
─────────────────────         ─────────────────────      ─────────────────────────
이상적 미래를 설계한다          현재 문제를 해결한다         결과를 검증하고 교훈을 축적한다
PRD → FS → IA → UF           분석 → 원인 → 해결 → 검증   수집 → 평가 → 반성 → 메모리화

         ┌──────────── 회고가 새 기획을 촉발 ──────────────┐
         ▼                                                 │
    [기획] ──→ [실행] ──→ [회고] ──→ [메모리 업데이트] ──→ [다음 사이클]
                              │
                              └──→ 문제 발견 시 problem-solver로 위임
```

## 핵심 철학 (반드시 내면화하라)

| 원칙 | 설명 | 근거 |
|------|------|------|
| **외부 접지(Grounding) 필수** | 순수 자기평가는 신뢰 불가. 반드시 외부 데이터(메트릭, 테스트, 사용자 피드백)를 포함 | CRITIC(ICLR 2024), Huang et al.(ICLR 2024) |
| **관찰이 아닌 행동 지침** | "소통 부족" ❌ → "ALWAYS: PR에 변경 이유 3줄 이상 작성" ✅ | ERL 논문: 휴리스틱이 궤적보다 일반화 우수 |
| **비난 없는 분석** | 사람이 아닌 시스템·프로세스에 초점. 실패는 학습 데이터 | Google SRE, AAR 메타분석(d=0.79) |
| **이중 루프 자동 트리거** | 반복 패턴 감지 시 "가정 자체가 틀린 건 아닌가?" 질문 삽입 | Argyris 이중 루프 학습 |
| **메모리 복리 효과** | 메타 규칙(규칙을 작성하는 규칙)으로 품질이 시간에 따라 복리로 향상 | claude-meta, AutoDream 패턴 |
| **최소 유효 회고** | 5분 마이크로 회고부터 반일 딥 회고까지 규모에 맞게 조절 | Agile Retro 실전 |

## Phase 0: 회고 유형 진단 (Gate & Classify)

요청을 받으면 **반드시 먼저** 아래 분류로 회고 유형을 판별하라.

### 회고 유형 분류표

| 유형 | 트리거 상황 | 소요 시간 | 산출물 |
|------|-----------|----------|--------|
| **M. 마이크로 회고** | 태스크 1개 완료, 짧은 작업 종료 | 3~5분 | DAKI 요약 + 메모리 항목 1~3개 |
| **S. 스프린트 회고** | 스프린트/주간 종료, 중간 점검 | 15~30분 | 5단계 분석 + 액션 아이템 + 메모리 업데이트 |
| **P. 프로젝트 회고** | 프로젝트 마일스톤/완료, 월간 리뷰 | 30~60분 | 전체 보고서 + 이중루프 분석 + 메모리 대정리 |
| **I. 인시던트 회고** | 장애/실패/중대 버그 발생 | 15~30분 | 타임라인 분해 + 5-Whys + COE 형식 |
| **C. 크로스 프로젝트 회고** | 여러 프로젝트 간 패턴 분석 | 30~60분 | 메타 분석 + 공통 패턴 + 조직 수준 교훈 |

### 자동 진단 로직

```
사용자 요청 분석:
├── "이 작업 어땠어?" / 짧은 리뷰 → M (마이크로)
├── "이번 주/스프린트 회고" → S (스프린트)
├── "프로젝트 전체 리뷰" / "월간 정리" → P (프로젝트)
├── "왜 이게 실패했지" / "장애 분석" → I (인시던트)
├── "KOR이랑 GO 프로젝트 비교" / "전체적으로 보자" → C (크로스)
└── 불명확 → 사용자에게 역질문 (최대 2개)
```

## Phase 1: 신호 수집 (Data Collection)

회고 유형이 결정되면, 해당 유형에 맞는 데이터를 수집한다.

### 1.1 자동 수집 가능한 신호

| 신호 유형 | Claude.ai Chat | Claude Code | 수집 방법 |
|----------|---------------|-------------|----------|
| 대화 이력 | ✅ 현재 대화 | ✅ 세션 로그 | 대화 턴 분석 |
| 과거 대화 | ✅ conversation_search | ✅ git log | 도구 사용 |
| 파일 변경 | ⚪ 업로드 파일 | ✅ git diff | 도구 사용 |
| 테스트 결과 | ⚪ 사용자 제공 | ✅ 테스트 출력 | 자동 수집 |
| 메모리 현황 | ✅ memory_user_edits view | ✅ CLAUDE.md | 도구 사용 |
| 프로젝트 지식 | ✅ project_knowledge_search | ✅ 파일 시스템 | 도구 사용 |

### 1.2 사용자에게 질문할 신호 (유형별)

**M (마이크로)**: 최소 질문 — "이 작업의 결과에 만족하나요? (1~5점)"

**S (스프린트)**:
1. 이번 스프린트에서 가장 큰 성과는?
2. 가장 큰 병목/장애물은?
3. 예상과 다르게 진행된 것은?

**P (프로젝트)**:
1. 초기 목표 대비 달성도는? (%)
2. 가장 자랑스러운 결정은?
3. 다시 한다면 바꾸고 싶은 것은?
4. 팀/도구/프로세스에서 배운 것은?

**I (인시던트)**:
1. 언제 문제를 인지했나?
2. 첫 대응까지 얼마나 걸렸나?
3. 근본 원인은 무엇이라 생각하나?

## Phase 1.5: 판단 분류기 (v2.5 Growth Engine)

회고 과정에서 발견한 각 교훈을 **판단(Judgment)** vs **실행(Execution)** 으로 분류한다.

| 분류 | 정의 | 예시 | 개선 방향 |
|------|------|------|----------|
| **판단 오류** | 잘못된 방향 선택. "무엇을 할지" 결정 실패 | 잘못된 라이브러리 선택, 불필요한 기능 구현 | unbounded-engine 메타질문 강화 |
| **실행 오류** | 올바른 방향이나 구현 실패. "어떻게 할지" 실패 | 타입 에러, 테스트 누락, 설정 오류 | problem-solver + ralph-loop |

**분류 규칙:**
- DAKI 결과의 Drop/Improve 항목마다 판단/실행 태그 부여
- 판단 오류 비율 > 30% → "사고 레이어를 더 강화하세요" 제안
- 실행 오류만 반복 → "코딩 도구 설정 점검하세요" 제안
- 분류 결과를 CLAUDE.md 교훈에 태그로 반영: `[판단]` / `[실행]`

## Phase 1.6: SCP 실패 분류기 (Service Completion Problem)

회고 과정에서 발견한 각 실패/개선 항목에 SCP 태그를 부여한다.

| 태그 | 질문 | 예시 |
|------|------|------|
| SCP-1 | 작업이 끝나기 전에 "완료"라고 했는가? | "3개 중 2개만 구현 후 완료 선언" |
| SCP-2 | 컨텍스트가 차면서 초반 결정을 잊었는가? | "초반에 결정한 API 구조와 다르게 구현" |
| SCP-3 | 어려운 부분을 슬쩍 빼고 쉬운 것만 했는가? | "에러 처리를 '추후 추가'로 연기" |
| SCP-4 | 직접 결정할 수 있는데 사용자에게 물었는가? | "파일명을 어떻게 할까요?" |
| SCP-5 | 검증했다고 했지만 실제로 안 했는가? | "테스트 통과" 주장인데 bash 이력 없음 |

### SCP 빈도 추적

```
세션 N 결과:
  SCP-1: 0건 | SCP-2: 1건 | SCP-3: 2건 | SCP-4: 0건 | SCP-5: 0건
  → 최다 유형: SCP-3 (복잡도 회피)
  → 제안: service-completion-checklist.md SCP-3 방어 로직 강화
```

## Phase 1.7: 자동 스킬 후보 제안 (Skill Candidate Extraction)

회고에서 누적된 교훈이 일정 조건을 충족하면 **신규 스킬 후보**를 제안한다.
단, **제안만 수행하며 스킬 생성은 사용자 명시 승인 후**에만 이루어진다.

### 트리거 조건 (AND)

1. 동일 주제의 교훈이 `rules/accumulated-lessons.md`에 **3회 이상** 축적
2. 동일 SCP 태그를 공유 (예: `[SCP-5]` 3건 모두 일치)
3. 기존 스킬 중 해당 주제를 커버하는 것이 없음 (description grep)

### 제안 템플릿

```
## 스킬 후보 제안 — {날짜}

이름: {kebab-case-name}
근거 교훈 (3건):
  - [SCP-N] {교훈 1} [since:YY/MM/DD]
  - [SCP-N] {교훈 2} [since:YY/MM/DD]
  - [SCP-N] {교훈 3} [since:YY/MM/DD]
목적: {1문장}
승인 시 생성 경로: skills/{name}/SKILL.md

❓ 생성하시겠습니까? (사용자 승인 필요 — 자동 생성 금지)
```

### 금지 사항

- **자동 생성 금지**: 후보 표시만. 사용자가 "생성해" 라고 명시해야 실제 파일 작성
- **중복 생성 금지**: 기존 스킬 description과 60% 이상 겹치면 "기존 X 스킬 확장" 으로 대체 제안
- 삭제/병합 제안은 별도 회고 사이클에서 (`meta-eval-log.md` 참조)

## Phase 2: 구조화된 평가 (Grounded Evaluation)

### 2.1 DAKI 프레임워크 적용 (모든 유형 공통)

수집된 데이터를 **DAKI** 4분류로 구조화한다:

```
┌─────────────────────────────────────────────────┐
│                    DAKI 분석                       │
├──────────────┬──────────────────────────────────┤
│ 🗑️ Drop      │ 폐기할 것 — 효과 없었던 전략/도구/습관   │
│ ➕ Add       │ 도입할 것 — 이번에 없었지만 필요했던 것   │
│ ✅ Keep      │ 유지할 것 — 잘 작동한 전략/도구/습관     │
│ 🔧 Improve   │ 개선할 것 — 방향은 맞지만 조정 필요      │
└──────────────┴──────────────────────────────────┘
```

### 2.2 유형별 추가 분석

**S/P 유형 — Spotify Health Check 차원 (해당 시)**:

| 차원 | 🟢 좋음 | 🟡 보통 | 🔴 나쁨 |
|------|---------|---------|---------|
| 미션 명확성 | 왜 하는지 안다 | 대략 안다 | 모르겠다 |
| 진행 속도 | 빠르게 전달 | 적당함 | 병목 있음 |
| 기술 품질 | 자랑스럽다 | 괜찮다 | 부끄럽다 |
| 학습 | 많이 배운다 | 가끔 배운다 | 정체됨 |
| 재미 | 즐겁다 | 보통이다 | 지친다 |

**I 유형 — Meta 6단계 타임라인 분해**:

```
감지(Detection) → 에스컬레이션(Escalation) → 완화(Mitigation)
→ 복구(Remediation) → 검증(Verification) → 종료(Close)

각 단계의 소요 시간과 병목 식별
```

### 2.3 이중 루프 자동 트리거 조건

아래 패턴이 감지되면 **자동으로 이중 루프 질문을 삽입**하라:

| 감지 패턴 | 삽입 질문 | 루프 수준 |
|----------|----------|----------|
| 동일 문제 3회 이상 반복 | "이 문제가 반복되는 근본 구조는 무엇인가?" | 이중 루프 |
| Drop 항목이 이전 회고의 Add 항목 | "왜 도입한 것이 효과가 없었나? 가정이 틀린 건 아닌가?" | 이중 루프 |
| 모든 차원이 🟡 | "현재 프레임워크 자체를 바꿔야 하는 건 아닌가?" | 삼중 루프 |
| 목표 달성했지만 허탈함 | "목표 자체가 올바른 것이었나?" | 삼중 루프 |

## Phase 3: 반성문 생성 (Verbal Reflection)

### 3.1 행동 지침 변환 규칙

모든 교훈은 반드시 **ALWAYS/NEVER/WHEN...THEN** 형식의 행동 가능한 지침으로 변환한다.

**변환 예시:**

| ❌ 관찰 (금지) | ✅ 행동 지침 (필수) |
|---------------|-------------------|
| "테스트가 부족했다" | "ALWAYS: PR 전에 최소 1개 엣지케이스 테스트 추가" |
| "소통이 안 됐다" | "WHEN: 설계 변경 시 THEN: 변경 이유를 CLAUDE.md에 즉시 기록" |

## Phase 4.5: 졸업 판정 (v2.5 Growth Engine)

교훈의 생명주기 단계를 자동 판정하고 승격/졸업을 제안한다.

### 교훈 생명주기

```
생성 → 축적(CLAUDE.md) → 패턴탐지(3회+) → 승격(rules/) → 강제화(hooks) → 졸업(archive)
```

### 졸업 판정 기준

| 단계 전환 | 조건 | 자동 제안 |
|-----------|------|----------|
| CLAUDE.md → rules/ | 같은 교훈 3회+ 반복 축적 | "이 교훈을 rules/로 승격하시겠습니까?" |
| rules/ → hooks | rules 위반이 2회+ 반복 | "이 규칙을 hooks로 강제화하시겠습니까?" |
| hooks → archive | 6개월간 위반 0회 | "이 훅은 내면화 완료. 아카이브하시겠습니까?" |

### 졸업 실행

```
승격 시:
1. CLAUDE.md에서 해당 교훈 제거 (중복 방지)
2. rules/{주제}.md에 추가
3. git commit -m "retro: 교훈 승격 → rules/{주제}.md"

강제화 시:
1. rules/ 규칙을 hooks/ 스크립트로 변환
2. settings.json에 hook 등록
3. git commit -m "retro: 규칙 강제화 → hooks/{이름}.sh"
```

## Phase 5: 검색 및 적용 (Retrieval & Application)

### 5.1 다음 세션 시작 시 자동 참조

다음 작업 시작 시, 관련 과거 회고를 자동 검색:

- 해당 프로젝트의 최근 DAKI 결과와 액션 아이템 요약 제공

### 5.1.5 복리 지수 산출 (v2.5 Growth Engine)

회고 완료 시 아래 복리 지수를 산출하여 시스템 성장률을 추적한다.

```
복리 지수 = (교훈→규칙 전환율) × (규칙→훅 전환율) × (재발 방지율)
         = (rules 승격 수 ÷ 3회+ 반복 교훈 수)
           × (hooks 강제화 수 ÷ rules 위반 반복 수)
           × (1 - 같은 에러 재발 수 ÷ 전체 에러 수)

목표: 0.75 이상 (6개월 내)
현재: session-briefing.sh에서 간이 측정
```

**출력 예시:**
```
📊 복리 지수: 0.62 (목표: 0.75)
  교훈→규칙: 4/7 = 0.57
  규칙→훅: 2/3 = 0.67
  재발방지: 18/20 = 0.90
  💡 교훈 승격을 3건 더 하면 0.75 도달 예상
```

### 5.2 연속 사이클 연결

```
회고 완료 후 자동 제안:
├── Add 항목이 기획 수준 → "planning-generator로 기획하시겠습니까?"
├── Drop/Improve가 버그/장애 → "problem-solver로 해결하시겠습니까?"
├── 반복 패턴 발견 → "ralph-loop으로 자동 최적화하시겠습니까?"
└── 메모리 대량 변경 → "메모리 정리 세션을 진행하시겠습니까?"
```

## 참고 자료

더 상세한 프레임워크와 사례는 `references/frameworks.md`를 참조하라.
이 파일에는 DAKI 상세 가이드, AAR 4대 질문 템플릿, Amazon COE 형식,
Spotify Health Check 전체 차원, 메모리 메타 규칙 상세가 포함되어 있다.

## DON'T
- "다음엔 잘하겠다" 같은 추상적 교훈 — ALWAYS/NEVER로 변환 안 되면 교훈 아님
- 회고 없이 다음 스프린트 시작 — 같은 실수 반복의 근본 원인
- 교훈을 문서에만 기록 — CLAUDE.md/rules/hooks에 미반영 = 복리 효과 없음

## DO
- git log --oneline으로 사실 기반 수집 먼저
- ALWAYS/NEVER를 즉시 CLAUDE.md에 반영
- 정량 데이터(시간/라인/Pass율) 우선, 감정/인상은 보조
