## 1. DAKI 프레임워크 상세

DAKI는 AI 에이전트에 가장 직접적으로 매핑되는 회고 프레임워크다.
모든 카테고리가 즉시 실행 가능한 행동으로 변환된다.

### 🗑️ Drop (폐기)
**질문**: "더 이상 하지 말아야 할 것은?"
**변환 형식**: `NEVER: [구체적 행동]. 이유: [근거]`
**예시**:
- NEVER: Pine Script에서 하드코딩된 SL% 사용. 이유: ATR 가변 SL이 일관성 0.81로 우월 (26/03/25 검증)
- NEVER: QuestDB 오프라인 상태에서 E2E 테스트 실행. 이유: "pass" 판정이 미검증 (26/03 하네스 발견)

### ➕ Add (도입)
**질문**: "이번에 없었지만 다음에 필요한 것은?"
**변환 형식**: `ADD: [구체적 행동]. 기대 효과: [예상 개선]`
**예시**:
- ADD: 전략 백테스트에 WFE(Walk-Forward Efficiency) ≥ 0.5 게이트 추가. 기대: 과적합 전략 조기 폐기
- ADD: 품의서 제출 전 호렌소 사전진단 실시. 기대: 반려율 50% 감소

### ✅ Keep (유지)
**질문**: "계속해야 할 것은?"
**변환 형식**: `KEEP: [구체적 행동]. 근거: [정량 데이터]`
**예시**:
- KEEP: S14 전략(P⅓ 지지) — PF 1.49, MDD 3.4%, BTC 검증 완료
- KEEP: 연구프로토콜 v2.0의 5탭 시스템. 근거: 가설→검증 사이클 시간 40% 단축

### 🔧 Improve (개선)
**질문**: "방향은 맞지만 어떻게 더 나아질 수 있을까?"
**변환 형식**: `IMPROVE: [현재 상태] → [목표 상태]. 방법: [구체적 접근]`
**예시**:
- IMPROVE: 하네스 E2E 테스트(현재 5건) → 15건. 방법: 에지케이스 시나리오 10건 추가
- IMPROVE: 품의서 견적검토(현재 수동 비교) → 자동화. 방법: estimate-comparison-generator 활용

## 2. AAR 4대 질문 템플릿

미 육군 After Action Review의 4대 질문 (효과 크기 d=0.79):

```
1. 무엇이 일어나야 했나? (의도/계획)
   → 초기 목표, 예상 결과, 성공 기준을 명시

2. 실제로 무엇이 일어났나? (현실)
   → 사실에 기반한 타임라인, 데이터, 결과 기술
   → 주관적 판단 배제, 관찰 가능한 사실만

3. 왜 차이가 있었나? (분석)
   → 시스템/프로세스/도구/환경 요인 분석
   → ❌ "누가 실수했다" → ✅ "이 프로세스에 이런 취약점이 있었다"

4. 무엇을 배울 수 있나? (교훈)
   → ALWAYS/NEVER/WHEN...THEN 형식으로 변환
   → 메모리에 기록할 항목 도출
```

### AAR 촉진 규칙
- 모든 참가자가 발언해야 함 (AI 회고에서는 "모든 관점 검토")
- 활동 직후에 실시 (기억이 신선할 때)
- 비난 금지 (시스템 중심 분석)
- 구체적 행동 지침 도출 필수

## 4. Spotify Health Check 전체 11차원

Henrik Kniberg가 개발, 5,000회 이상 실시된 검증 프레임워크:

| # | 차원 | 🟢 우수 | 🔴 열악 |
|---|------|---------|---------|
| 1 | 릴리스 용이성 | 쉽게 릴리스 가능 | 릴리스가 고통 |
| 2 | 프로세스 적합성 | 프로세스가 우리에게 맞음 | 프로세스가 방해 |
| 3 | 기술 품질 | 코드에 자부심 | 기술 부채 심각 |
| 4 | 가치 | 우리가 만드는 것이 가치 있음 | 왜 하는지 모름 |
| 5 | 속도 | 빠르게 일함 | 끝없이 대기 |
| 6 | 미션 | 왜 존재하는지 알고 열정 | 미션 불명확 |
| 7 | 재미 | 출근이 즐거움 | 지루하고 지침 |
| 8 | 학습 | 계속 배움 | 정체됨 |
| 9 | 지원 | 필요한 지원을 받음 | 방치됨 |
| 10 | 자율성 | 스스로 결정 | 마이크로 관리됨 |
| 11 | 팀워크 | 협업이 좋음 | 사일로 |

### 트래픽 라이트 + 트렌드 화살표 표기법
- 🟢↑ : 좋고 개선 중
- 🟢→ : 좋고 유지
- 🟡↓ : 보통이고 악화 중 ← **주의 필요**
- 🔴↓ : 나쁘고 악화 중 ← **즉시 대응**

## 5. 5-Whys 구조화 템플릿

```
문제 정의: [관찰 가능한 증상을 구체적으로]

Why 1: 왜 [증상]이 발생했는가?
→ 답: [직접 원인]
→ 증거: [로그/데이터/관찰]

Why 2: 왜 [직접 원인]이 존재했는가?
→ 답: [프로세스 원인]
→ 증거: [...]

Why 3: 왜 [프로세스 원인]이 방지되지 않았는가?
→ 답: [시스템 원인]
→ 증거: [...]

Why 4: 왜 [시스템 원인]이 존재하는가?
→ 답: [구조적 원인]
→ 증거: [...]

Why 5: 왜 [구조적 원인]이 개선되지 않았는가?
→ 답: [근본 원인] ← 여기서 행동 지침 도출
→ 행동 지침: ALWAYS/NEVER/WHEN...THEN

⚠️ 주의: 5번째 Why에서 "사람이 실수했다"가 나오면 4번째로 돌아가서
  "왜 그 실수가 가능한 시스템이었는가?"를 다시 물어라.
```

## 6. 메모리 메타 규칙 상세

### 메모리 작성 품질 기준

| 기준 | 좋은 예 | 나쁜 예 |
|------|--------|---------|
| 간결성 | "ALWAYS: PR에 변경 이유 3줄 작성" | "팀에서 PR을 작성할 때 변경 이유를 충분히 설명하는 것이 좋다고 합의했으며..." |
| 구체성 | "SL = ATR(20)×5, 일관성 0.81" | "적절한 손절선 설정" |
| 검증 가능성 | "PF > 1.5 + WFE ≥ 0.5 게이트" | "좋은 성과를 내는 전략만 사용" |
| 날짜 포함 | "26/04/03 확인: ..." | 날짜 없음 |
| 프로젝트 태그 | "[GO] KEEP: S14..." | 태그 없음 |

### 메모리 정리 주기

| 주기 | 작업 |
|------|------|
| 매 회고 시 | 신규 항목 추가, 직접 관련 항목 업데이트 |
| 월 1회 | 전체 메모리 스캔, 중복 병합, 모순 해결 |
| 분기 1회 | 아카이브 대상 식별, 메타 규칙 업데이트 |

### 충돌 해결 프로토콜

```
메모리 항목 충돌 감지 시:
1. 두 항목의 날짜 확인 → 최신이 우선
2. 두 항목의 근거 확인 → 정량 데이터가 정성 판단보다 우선
3. 해결 불가 시 → 사용자에게 직접 확인 (둘 다 보여주고 선택 요청)
4. 해결 후 → 병합된 단일 항목으로 교체
```

## 7. 이중/삼중 루프 학습 가이드

### 단일 루프 (기본 — 대부분의 회고)
**질문**: "올바르게 하고 있는가?"
**초점**: 행동 수정 — 같은 목표, 더 나은 방법
**예시**: "SL을 고정%에서 ATR 가변으로 바꾸자"

### 이중 루프 (심화 — 반복 패턴 감지 시)
**질문**: "올바른 일을 하고 있는가?"
**초점**: 가정/목표 자체 재검토
**예시**: "15분봉 차트에서 계속 실패하는 건, 15분봉이 이 전략에 부적합한 건 아닌가?"

### 삼중 루프 (전환적 — 모든 차원이 정체 시)
**질문**: "왜 이 프레임워크가 올바르다고 생각하는가?"
**초점**: 사고 방식/패러다임 자체 전환
**예시**: "P이론 자체가 아닌 완전히 다른 접근법이 필요한 건 아닌가?"
