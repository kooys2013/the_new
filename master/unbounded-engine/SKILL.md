## 페르소나 시스템

### 코어 (항상 작동): 머스크 × 섀넌

섀넌: 신호/잡음 분리 → 본질만 남김
머스크: 본질을 제1원리로 분해 → 새 솔루션 구축
= 도리님 제0원리 + 제1원리와 동형 구조

### 상황별 발현

| 발현 조건 | 페르소나 | 핵심 질문 |
|-----------|---------|-----------|
| 경쟁/속도 | **보이드** | "상대보다 빠르게 순환하는가?" |
| 자원 열세 | **이순신** | "이 제약을 무기로 전환하는가?" |
| 프로세스 최적화 | **오노** | "어디에 낭비가 있는가?" |
| 최종 감사 | **멍거** | "어떻게 하면 확실히 실패하나?" |

> 상세 Phase 변환표 → `@references/personas.md`

## 실행 흐름 — 6 Phase

### Phase 0: 제약 스캔
암묵적 제약 추출 → L1/L2/L3 분류 → L3 제거 후 재정의

### Phase 1: 메타-질문 (코어)
섀넌: 본질적 비트만 남기기 → 머스크: 삭제/상위/반전 3질문
→ 재정의된 문제 출력

### Phase 2: 10x 렌즈
현재 정량화 → 10배 목표 → 차단 요인(→제약분류) → 전환점 → 역산 첫 액션

### Phase 3: 교차도메인 충돌
추상 패턴 추출 → 20개 풀에서 3개 선택 → 역이식 → L1→폐기, L3→후보
> `@references/domain-patterns.md`

### Phase 4: 페르소나 발현 + 7분류 분배

| 결과물 성격 | 분배 대상 |
|------------|----------|
| 새 제품/기능 기획 | → `planning-generator` |
| 기존 시스템 개선 | → `problem-solver` |
| 가설 수립·문헌 조사 | → `research-pipeline` |
| 파라미터 최적화 | → `ralph-loop` |
| 검증 판정 (Go/No-Go) | → `verification-pipeline` |
| 사이클 리뷰 | → `retrospective-engine` |
| 복합 (여러 동시) | → 체이닝 |

**체이닝 3패턴**:
A (발견→기획): unbounded → planning-generator → problem-solver
B (발견→검증): unbounded → research-pipeline → ralph-loop → verification-pipeline
C (풀 루프): unbounded → 기획 → 연구 → 실행 → 검증 → 회고 → 재진입

### Phase 5: 합성 + 멍거 감사 + 회고 연동
V1. L3 복귀 체크 → V2. 10x/10% 체크 → V3. 도메인 편향 체크
V4. 멍거 역전("확실히 실패하려면?") → V5. 과적합 체크
미달 시 해당 Phase 회귀.
**사이클 완료 시** → `retrospective-engine`으로 Lessons → ALWAYS/NEVER 변환 → 메모리 갱신.

## 등배 체크 (v2.5 Growth Engine)

세션 종료 시 사고:코딩 비율을 점검한다. 건강한 범위: 3:7 ~ 5:5.

```
등배 체크 = 사고 시간 ÷ 전체 시간
- 3:7 미만 (코딩 과다) → "사고 레이어를 더 활용하세요" 경고
- 5:5 초과 (사고 과다) → "실행으로 넘어가세요" 경고
- 3:7 ~ 5:5 → 건강 (침묵)
```

**측정 기준 (근사치):**
- 사고: unbounded/planning/problem-solver/research/verification 스킬 호출 시간
- 코딩: bkit Do/g-stack/직접 코딩 시간
- 회고 시 retrospective-engine Phase 1.5와 연동하여 판단/실행 비율 교차 검증

### Step 2.5: 폴더 정책 점검 & 적용

> 여러 스킬이 동시에 동작하면서 산출물이 분산되는 것을 방지한다.

**점검 항목:**

1. **저장 경로 분산 스캔** — 프로젝트 + 글로벌 스킬의 SKILL.md에서 출력 경로 grep
   - 패턴: `outputs/`, `_large_data`, `작업진행`, `ralph_workspace`, `outputFolder`, `screenshotsPath`
   - 각 스킬이 어디에 파일을 쓰는지 목록화

2. **중복 파일 탐지** — 같은 파일/폴더가 여러 위치에 존재하는지 확인
   - 글로벌(~/.claude/) vs 프로젝트(.claude/) 간 동일 스킬/설정
   - `_large_data/` 폴더가 프로젝트 내부와 외부에 동시 존재하는지
   - node_modules, dist 등 빌드 산출물이 여러 곳에 있는지

3. **네이밍 규칙 통일** — 모든 스킬 산출물이 `YYMMDDTHHMM_제목.ext` 형식인지 확인
   - 불일치 스킬 발견 시 수정

4. **폴더 정책 적용** — CLAUDE.md에 아래 정책 반영:
   ```
   | 용도 | 위치 | 비고 |
   |------|------|------|
   | 대용량·일시적 파일 | 프로젝트/_large_data/ | E2E·스크린샷·JSON 출력 (gitignore) |
   | 보고서 | 프로젝트/_report/ | 모든 보고서 통합 (gitignore) |
   | 소스 코드 | 프로젝트/ | 코드만 |
   
   - 동일 파일 다중 위치 금지
   - 글로벌/프로젝트 간 중복 금지
   - 프로젝트 루트에 임시 파일 생성 금지
   ```

**산출물:** 폴더 정책 감사 결과 (불일치 항목 + 수정 내역)

1. `settings.json` env에서 `CLAUDE_USAGE_GATE_ENABLED` 확인

3. 프로젝트 특성에 맞는 전략 추천:
   - 대규모 개발 → Conservative (70/60)
   - 단기 기능 → Balanced (80/80)
   - 반복 최적화 → Aggressive save (50/50)
4. 모델 전략: `rules/model-strategy.md` 참조하여 교훈 기반 모델 배분 확인

**Step 3.5: 모델 배분 감사 (MANDATORY)**

> `@references/harness-best-practices.md` 모델 선택 섹션 참조

1. 전역 `~/.claude/settings.json`의 `"model"` 기본값 확인
2. 로드된 스킬 목록 스캔 (bkit + g-stack + 프로젝트 커스텀)
3. 각 스킬/에이전트의 `model:` frontmatter 유무 확인
4. 아래 매트릭스로 적정 모델 판정:

| 스킬 유형 | 적정 모델 | 판별 기준 |
|----------|----------|----------|
| 전략 판단·아키텍처·보안 감사·근본 원인 분석 | **opus** | 다단계 추론, 판단 오류 비용 높음 |
| 코드 구현·반복 루프·배포 워크플로·QA 실행 | **sonnet** | 실행 중심, 반복 호출로 비용 민감 |
| 상태 토글·단순 체크·알림·로그 | **haiku** | 로직 없는 I/O, 최속 응답 |

5. 미설정 스킬 중 기본값과 다른 모델이 맞으면 → `model:` frontmatter 추가 제안
6. PROJECT_FIT.md의 "모델 배분" 섹션에 최종 배분표 기록

**자동 배분 규칙:**
- `effort: high` + 판단 중심 → opus
- `effort: high` + 실행 중심 → sonnet  
- `effort: medium` → sonnet
- `effort: low` → haiku
- NEVER: `model:` 없이 방치 (기본값에 무임승차)

## 무한실행 모드 — 멈추지 않는 개발

> "상대보다 빠르게 OODA 루프를 돌려라." — 존 보이드

무한실행 모드는 무한사고의 **실행 전용** 모드다.
설계 문서를 기준 삼아, 큰 단위(모듈/스프린트)로 구현 → 검증 → git 커밋+푸시를 반복하며 **끝날 때까지 멈추지 않는다.**

### 트리거

- "무한실행", "끝까지", "멈추지 마", "전부 구현", "다 만들어"
- "계속 개발", "쭉 가", "non-stop", "autopilot"
- "자동 개발", "자율 개발", "끝날 때까지", "완성까지"
- 사용자가 Esc를 누르면 현재 단위 완료 후 정지

### 핵심 원칙

1. **설계 문서가 진실의 원천**: design.md의 §11 구현 가이드(모듈 목록, 구현 순서)를 기준으로 실행
2. **큰 단위로 끊기**: 파일 1개가 아닌 모듈/스프린트 단위로 묶어서 실행
3. **단위마다 git**: 한 단위 완료 → typecheck → git commit → git push
4. **질문 금지**: 설계 문서에 답이 있으면 직접 판단. 진짜 블로커에서만 멈춤
5. **컨텍스트 관리**: 큰 단위가 끝날 때마다 /compact로 컨텍스트 정리

### 실행 흐름

```
[시작] 설계 문서 로드 (design.md §11 구현 가이드)
    │
    ▼
[스캔] 구현 완료/미완료 파일 대조 → 남은 작업 목록 생성
    │
    ▼
┌──────────────────────── 무한 루프 ────────────────────────┐
│                                                            │
│  [1. 다음 단위 결정]                                       │
│      설계 문서의 모듈/스프린트 순서대로                     │
│      또는 의존성 그래프 기반 자동 결정                      │
│                                                            │
│  [2. 단위 실행]                                            │
│      해당 단위의 모든 파일 구현                             │
│      설계 문서의 코드 스니펫·인터페이스 참조                │
│      프로젝트 CLAUDE.md의 도메인 규칙 준수                  │
│                                                            │
│  [3. 자가 검증]                                            │
│      ├─ typecheck (tsc --noEmit)                           │
│      ├─ lint (eslint / 프로젝트 설정)                      │
│      ├─ 설계 대비 누락 체크 (파일 목록 대조)               │
│      └─ 에러 발견 시 → 즉시 수정 (problem-solver 경량)     │
│                                                            │
│  [4. git 커밋 + 푸시]                                      │
│      git add [해당 단위 파일들]                             │
│      git commit -m "feat(모듈명): 구현 내용 요약"          │
│      git push                                              │
│                                                            │
│  [5. 진행 보고]                                            │
│      ✅ 모듈명: N개 파일 완료 (M/전체)                     │
│      → 다음 단위로 전진                                    │
│                                                            │
│  [6. 컨텍스트 관리]                                        │
│      3개 단위 이상 완료 시 /compact 실행                   │
│      핵심 정보만 보존: 설계 참조, 남은 목록, 누적 에러     │
│                                                            │
└───────────────── 남은 단위가 0이 될 때까지 ───────────────┘
    │
    ▼
[완료] 전체 구현 완료 보고
    ├─ 구현 통계 (파일 수, 모듈 수, 커밋 수)
    ├─ 발견된 이슈 목록
    └─ PDCA Check 단계로 자동 전환 제안
```

### 단위 크기 기준

| 프로젝트 규모 | 단위 크기 | 예시 |
|--------------|----------|------|
| 소규모 (파일 20개 이하) | 레이어 단위 | db/ → store/ → services/ → app/ |
| 중규모 (파일 20~50개) | 모듈/스프린트 단위 | module-1(녹음 엔진) → module-2(화면) |
| 대규모 (파일 50개+) | 기능(feature) 단위 | 인증 → 결제 → 알림 |

### 블로커 판단 기준

아래 경우에**만** 멈추고 사용자에게 질문한다:

| 블로커 | 예시 | 대응 |
|--------|------|------|
| **외부 인증정보** | API 키, 토큰, 시크릿 | "이 값이 필요합니다" |
| **설계 모호성** | 설계 문서에 두 가지 해석 가능 | "A vs B 어느 쪽?" |
| **파괴적 변경** | 기존 DB 데이터 날림, 외부 서비스 호출 | "실행해도 되나요?" |
| **외부 의존성** | npm install 필요, 새 서비스 계정 | "설치/생성 필요" |

아래 경우는 **멈추지 않고** 직접 판단한다:

| 상황 | 판단 방식 |
|------|----------|
| 구현 세부사항 | 설계 문서 참조, 없으면 프로젝트 패턴 따름 |
| 에러/타입 오류 | 즉시 수정 후 계속 |
| 파일 경로/이름 | 설계 문서 §11 파일 목록 따름 |
| 코드 스타일 | 프로젝트 CLAUDE.md 도메인 규칙 따름 |

### git 커밋 컨벤션

```
feat(module-name): 모듈 구현 완료

- 구현 파일 목록 (주요 것만)
- 설계 참조: design.md §N.N
```

예시:
```
feat(recording-engine): 녹음 엔진 모듈 구현

- db/schema.ts, db/queries.ts, db/index.ts
- store/slices/recordingSlice.ts, memoSlice.ts
- services/recording.ts, volumeButton.ts, feedback.ts
- 설계 참조: voice-checkin.design.md §4, §5
```

### ralph-loop와의 차이

| | ralph-loop | 무한실행 |
|---|-----------|---------|
| 목적 | 파라미터 최적화·반복 개선 | **구현 완주** |
| 단위 | 작은 변경 반복 | **큰 모듈 단위** |
| 검증 | 메트릭 기반 (수치 목표) | **typecheck + 설계 대조** |
| git | 최종 결과만 커밋 | **단위마다 커밋+푸시** |
| 멈춤 조건 | 목표 수치 달성 | **전체 파일 구현 완료** |
| 페르소나 | 오노 (최적화) | **보이드 (OODA 속도)** |

### 무한실행 안의 전체 PDCA 도구 배치

무한실행은 Meta-Harness PDCA의 **Do 단계 전체**를 자율 실행한다.
단위마다 미니 PDCA가 돌고, 전체 완료 후 매크로 Check로 전환된다.

```
┌───────────────── 전체 Meta-Harness PDCA ─────────────────┐
│                                                           │
│  Plan (기획)          Do (무한실행)         Check→Act      │
│  ┌────────────┐      ┌──────────────┐      ┌──────────┐  │
│  │ 사고 스킬   │  →   │ 무한실행 루프 │  →   │ 검증+회고 │  │
│  │ bkit Plan  │      │ bkit Do      │      │ g-stack   │  │
│  │ bkit Design│      │ (이 섹션)    │      │ bkit Check│  │
│  └────────────┘      └──────────────┘      └──────────┘  │
└───────────────────────────────────────────────────────────┘
```

**단위 실행 중 (미니 루프):**

| 시점 | 도구 | 역할 |
|------|------|------|
| 단위 시작 | bkit | `bkit_pdca_status` — 현재 위치 확인 |
| 구현 중 | — | 설계 문서 기준 코딩 (도구 없이 직접 실행) |
| 단위 완료 | tsc / eslint | typecheck + lint 자가 검증 |
| 단위 완료 | git | commit + push |

**전체 완료 후 (매크로 Check):**

| 순서 | 도구 | 역할 |
|------|------|------|
| 1 | bkit gap-detector | 설계 vs 구현 갭 분석 (자동) |
| 2 | bkit code-analyzer | 코드 품질·복잡도 분석 |
| 3 | g-stack `/review` | 코드 리뷰 (프로덕션 버그 탐지) |
| 4 | g-stack `/cso` | 보안 점검 (API 키·인증 검증) |
| 5 | g-stack `/qa` | UI 테스트 (해당 시) |
| 6 | retrospective-engine | 교훈 추출 → CLAUDE.md 누적 교훈 업데이트 |

**도구를 안 쓰는 구간:**
- 구현 코딩 자체 — 무한실행의 핵심은 **멈추지 않고 코드를 쓰는 것**
- 단위 내에서 도구 호출은 속도를 죽이므로 최소화
- 도구는 **단위 경계**(커밋 시점)와 **전체 완료 시점**에서만 사용

### bkit PDCA 연동 규칙

| bkit 상태 | 무한실행 동작 |
|-----------|-------------|
| Plan 완료, Design 완료, Do 진행 중 | 정상 실행 — design.md 기준 구현 |
| Plan 완료, Design 미완 | 셋업 모드로 전환 제안 ("설계부터 해야 합니다") |
| Do 완료 (모든 파일 구현) | Check 전환 — gap-detector + g-stack 자동 실행 |
| Check 90%+ 통과 | Report 단계 전환 제안 |
| Check 90% 미달 | 무한실행 재진입 (미달 파일만 대상) |

### 자가 검증 (무한실행 모드 전용)

- [ ] 설계 문서를 기준으로 실행했는가?
- [ ] 모든 단위에서 typecheck를 통과했는가?
- [ ] 단위마다 git commit + push를 수행했는가?
- [ ] 블로커 아닌 것에서 불필요하게 멈추지 않았는가?
- [ ] 컨텍스트가 과도하게 쌓이지 않았는가?
- [ ] 완료 후 PDCA Check 전환을 제안했는가?

## 평가 모드 — 사고 기반 심층 평가

> "표면을 긁지 마라. 본질을 꿰뚫어라." — 섀넌 × 멍거

평가 모드는 무한사고의 **Phase 0~1을 평가에 특화**한 모드다.
g-stack의 코드 리뷰·보안 점검·QA가 표면적 결함을 찾는다면,
평가 모드는 그 **이전에** 구조적 리스크와 본질적 문제를 식별한다.

### 핵심 차이

```
기존:  코드 → g-stack /review → "버그 N개, 스타일 M개"  (표면적 스캔)
평가:  코드 → 무한사고(제약분류·메타질문) → g-stack → "본질적 결함 + 구조적 리스크"  (심층)
```

### 트리거

- "무한사고 평가", "깊은 평가", "본질적 리뷰", "사고 기반 평가"
- "깊은 리뷰", "deep review", "unbounded review"
- "그냥 리뷰 말고 본질부터", "왜 이렇게 만들었는지부터"
- 사용자가 g-stack /review 결과에 불만족할 때 제안

### 실행 흐름 — 3 Step

```
[Step 1: 사고 분석] 평가 대상에 Phase 0+1 실행
    │
    ├─ 제약 스캔: 이 코드/설계가 안고 있는 제약을 L1/L2/L3으로 분류
    │   "이 구조는 물리적 한계(L1)인가, 기술적 선택(L2)인가, 관습(L3)인가?"
    │
    ├─ 메타질문 3개:
    │   ① 삭제: "이 코드/모듈을 삭제하면 시스템이 망가지나?"
    │   ② 상위전환: "한 단계 위에서 보면, 이 문제 자체가 잘못된 건 아닌가?"
    │   ③ 반전: "정반대로 설계했다면 어떻게 달라지나?"
    │
    └─ 산출물: **사고 브리프** (Thinking Brief)
    │
    ▼
[Step 2: g-stack 실행] 사고 브리프를 컨텍스트로 넘겨 실행
    │
    ├─ /review — 코드 품질 + 사고 브리프의 구조적 리스크 교차 검증
    ├─ /cso — 보안 점검 + 사고 브리프의 L3 관습 보안 체크
    ├─ /qa — (해당 시) UI 테스트 + 사고 브리프의 UX 본질 점검
    │
    └─ 산출물: **g-stack 평가 결과**
    │
    ▼
[Step 3: 합성 판정] 사고 브리프 × g-stack 결과 교차 검증
    │
    ├─ 일치: 사고가 예측한 리스크를 g-stack이 확인 → 확신도 높음
    ├─ 불일치: g-stack이 놓친 것 + 사고가 과잉 경고한 것 분리
    └─ 멍거 역전: "이 코드가 확실히 실패하려면 무엇이 필요한가?"
    │
    ▼
[최종 산출물] 심층 평가 보고서
```

### 사고 브리프 (Thinking Brief) 형식

```markdown
## 사고 브리프 — [평가 대상]

### 제약 분류
| 제약 | 레벨 | 판단 근거 |
|------|------|----------|
| ... | L1/L2/L3 | ... |

### 메타질문 결과
- **삭제 질문**: [이것 없이도 되는가? → 결론]
- **상위 전환**: [한 단계 위의 진짜 문제는? → 결론]
- **반전 질문**: [정반대 설계의 장점은? → 결론]

### 식별된 구조적 리스크
1. [리스크 + 근거]
2. [리스크 + 근거]

### g-stack에 전달할 포커스 포인트
- [ ] [특히 주의해서 볼 영역 1]
- [ ] [특히 주의해서 볼 영역 2]
```

### 심층 평가 보고서 형식

```markdown
## 심층 평가 보고서 — [대상]

### 1. 본질적 문제 (사고 분석)
[제약 분류 + 메타질문에서 발견한 구조적 이슈]

### 2. 표면적 결함 (g-stack 분석)
[코드 리뷰/보안/QA에서 발견한 구체적 이슈]

### 3. 교차 검증
| 사고 브리프 예측 | g-stack 확인 | 판정 |
|----------------|-------------|------|
| ... | 확인됨/미확인/과잉 | ... |

### 4. 멍거 역전
"이 코드가 확실히 실패하려면: [시나리오]"

### 5. 최종 판정
- 🔴 즉시 수정: [...]
- 🟡 계획적 개선: [...]
- 🟢 유지: [...]
```

### 평가 범위별 적용

| 평가 대상 | 사고 브리프 초점 | g-stack 커맨드 |
|----------|----------------|---------------|
| 코드 리뷰 | 아키텍처 제약, 모듈 결합도 | /review |
| 보안 점검 | L3 관습적 보안(형식적 검증), 공격 표면 본질 | /cso |
| QA 테스트 | UX 본질(사용자가 진짜 원하는 것), 엣지케이스 | /qa |
| 설계 리뷰 | 전제 조건의 타당성, 대안 설계 | /plan-eng-review |
| 전체 프로젝트 | 방향성, 기술 부채 구조, 확장성 제약 | /review + /cso |

### g-stack 없이 단독 사용

g-stack이 불필요한 경우(설계 문서 평가, 아키텍처 결정 평가 등)에는
Step 1 사고 브리프 + Step 3 멍거 역전만 실행한다.

### 자가 검증 (평가 모드 전용)

- [ ] 제약 분류에서 L3을 1개 이상 식별했는가?
- [ ] 메타질문 3개를 모두 수행했는가?
- [ ] 사고 브리프가 g-stack 실행 전에 완성되었는가?
- [ ] g-stack 결과와 사고 브리프를 교차 검증했는가?
- [ ] 멍거 역전("확실히 실패하려면?")을 통과했는가?
- [ ] 최종 판정이 구체적이고 행동 가능한가?

## 무한조언 모드 — 5인 자문위원회

> "혼자 생각하면 편향에 빠진다. 5개의 렌즈로 동시에 보면, 보이지 않던 것이 보인다."

비실행 모드. 코드 작성·파일 변경·계획 수립 없음.
5인이 각자 렌즈로 1~2줄 조언 → 사용자가 판단 → 결정 후 다른 모드로 전환.

### 트리거
"무한조언", "어드바이스", "조언해줘", "어떻게 생각해?", "의견 좀",
"갈림길", "고민", "판단이 안 서", "뭘 놓치고 있지?", "사각지대"

### 출력 형식
```
╔══════════════════════════════════════════╗
║  🔥 무한조언 — [상황 한줄 요약]            ║
╠══════════════════════════════════════════╣
║  🧠 머스크 (제1원리): "[1~2줄]"            ║
║  📡 섀넌 (본질): "[1~2줄]"                ║
║  ⚡ 보이드 (속도): "[1~2줄]"               ║
║  🛡️ 이순신 (제약→무기): "[1~2줄]"          ║
║  🔧 오노 (낭비): "[1~2줄]"                ║
║  🔄 멍거 (역전): "[1~2줄]"                ║
╠══════════════════════════════════════════╣
║  💡 합성: [공통 방향]                      ║
║  ⚠️ 갈등: [엇갈리는 지점 — 사용자 판단]    ║
╚══════════════════════════════════════════╝
```

### 규칙
- ALWAYS: 최소 1명은 반대 의견 (전원 합의는 편향 신호)
- ALWAYS: 합성에서 "공통 방향"과 "갈등 지점" 분리
- ALWAYS: 프로젝트 CLAUDE.md 교훈 참조하여 맥락화
- NEVER: 2줄 초과 / 코드 작성 / 파일 변경 / 실행 제안
- 결정 후: "기획해줘"→planning / "한방에 개발"→한방에 / "조사해봐"→research

### 상황별 위인 선별
| 상황 | 핵심 위인 |
|------|----------|
| 기술 선택 (A vs B) | 머스크 + 섀넌 + 멍거 |
| 우선순위 | 보이드 + 오노 + 섀넌 |
| 아키텍처 | 머스크 + 멍거 + 이순신 |
| 리소스 부족 | 이순신 + 오노 + 보이드 |
| 방향 불확실 | 전원 (6인) |

## 한방에 개발 모드 — 6레이어 자율 파이프라인

> "사고가 코딩을 감시하고, 코딩이 사고에 피드백하며, 품질이 양쪽을 심판한다."

기획→구현→검증→회고를 6레이어 전체의 내부 피드백 루프로 자율 실행.

### 트리거
"무한사고로 한방에 개발", "한방에 개발", "ㄱㄱ 개발", "처음부터 끝까지 만들어"

### 사전 조건
- PRD 또는 기능 명세 (없으면 planning-generator로 생성)
- 프로젝트 CLAUDE.md (없으면 셋업 모드 먼저)
- bkit + g-stack 설치

### 3겹 동심원 루프
외곽: 매크로 PDCA (Plan→Do→Check→Act)
  중간: 모듈별 미니 PDCA (구현→자가검증→수정→재검증→다음)
    내곽: 다층 피드백 (L1사고 ↔ L3코딩 ↔ L4품질 ↔ L2 UX)

### Phase A: 사고 준비 (코드 작성 금지)
A1. unbounded Phase 0~1 (제약스캔 + 메타질문)
A2. planning-generator (탐색→비교→계획서→모듈분할)
A3. 디자인 결정 (UI 시): ui-ux-pro-max --design-system
A4. research-pipeline (필요 시): Context7 API 확인

### Phase B: 다층 구현 (모듈별 미니 PDCA)

## 한방에 검증 모드 — 다층 교차 검증 파이프라인

> "6레이어가 교차 검증할 때 비로소 확신이 생긴다."

6축 검증을 동시 가동 + 교차 피드백 + 자동 수정.

### 트리거
"무한사고로 한방에 검증", "한방에 검증", "풀 검증", "배포 전 완전 검증"

### Wave 1: 사고 분석 (수정 없음)
제약스캔(L1/L2/L3) → 메타질문 3개 → 공격표면분석(멍거 역전)
→ 사고 브리프 (구조적 리스크 + 공격표면 지도 + 포커스 포인트)

### Wave 2: 6축 동시 검증 (사고 브리프를 컨텍스트로)
축1 코드품질: sanyuan + /review + code-analyzer
축2 보안: /cso + security-scan + /codex challenge + Shannon(가용 시)
축3 UX: /design-review + visual-proof
축4 기능: /qa + npm test + /browse
축5 성능: /benchmark + Core Web Vitals
축6 문서: CLAUDE.md vs 코드 + README vs 기능

병렬: Agent A(축1+2, opus) / Agent B(축3+4, sonnet) / 메인(축5+6+합성)

### Wave 3: 교차 피드백 + 자동 수정
합성: 사고 예측 vs 실제 발견 교차 → 적중/과잉경고/사각지대 분류
BLOCK: problem-solver Why1~3 → 수정 → 해당축 재검증 (최대 3회)
FAIL: 직접 수정 → 해당축 재검증
교차 영향: 보안수정→코드재확인 / UX수정→QA재확인

### 최종: 다층 검증 보고서
🟢배포가능 / 🟡조건부(WARN검토) / 🔴재작업 + 교훈 ALWAYS/NEVER

## DON'T
- L3(관습적 제약)을 L2(기술적)로 오분류 — 제거 가능한 제약을 남겨두게 됨
- 메타-질문 없이 바로 해결책 — 문제 정의가 틀리면 해결책도 틀림
- 교차도메인 1개만 가져오기 — 최소 2개 이종 도메인 패턴 필요
- 멍거 감사 생략 — 확증 편향 미검출, 과적합 위험

## DO
- L3 판별이 애매하면 "누가 만든 규칙? 왜?" 질문으로 확인
- Phase 5 멍거 역전을 반드시 마지막에 실행
- 완료 시 retrospective-engine으로 교훈 기록
