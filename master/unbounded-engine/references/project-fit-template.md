## 모델 배분 (스킬/에이전트 frontmatter)

> 전역 기본값(settings.json `model`)을 기준으로 오버라이드가 필요한 것만 명시.
> 기본값이 opus면 sonnet/haiku만 적어도 된다.

| 스킬/에이전트 | 배정 모델 | 이유 |
|--------------|----------|------|
| (예: unbounded-engine) | opus | 전략 판단·다단계 추론 |
| (예: ralph-loop) | sonnet | 수십 번 반복 호출 → 비용 |
| (예: checkpoint) | haiku | 상태 저장·읽기만 |

**판별 체크리스트 (셋업 모드 Step 3에서 실행):**
- [ ] 전략 판단 / 근본 원인 / 보안 감사 → opus
- [ ] 코드 구현 / 반복 루프 / 배포 워크플로 → sonnet
- [ ] 상태 토글 / 단순 체크 / 알림 → haiku
- [ ] 전역 기본값 확인 후 불필요한 중복 제거

## 셋업 프로세스 — 자동 채움 가이드

템플릿의 `[placeholder]`를 채울 때 아래 순서로 스캔한다:

### Step 1: 프로젝트 스캔 (자동)
```
1. package.json / build.gradle / Cargo.toml / go.mod / requirements.txt 등 → 스택 추출
2. README.md → 프로젝트 설명
3. .git/HEAD + git log --oneline -5 → 현재 단계 추정
4. src/ or app/ 구조 → 아키텍처 패턴 식별
5. .claude/skills/ → 기존 도메인 스킬 목록
6. CLAUDE.md 기존 여부 → 업데이트 vs 신규 생성 판단
```

### Step 2: 도메인 규칙 추론 (반자동)
```
스캔 결과 기반으로 초안 생성 → 사용자 확인 요청
- 금융/의료: ALWAYS 보안·감사 로깅, NEVER 민감 데이터 로그
- API 서버: ALWAYS 에러 핸들링 패턴, WHEN 성능 병목 THEN 프로파일링
- 프론트엔드: ALWAYS 접근성, WHEN 상태 복잡 THEN 상태관리 검토
```

### Step 3: 사고 흐름 비중 매핑 (반자동)
```
프로젝트 단계별 기본 프로파일:
- MVP:         기획(높음) > 문제해결(보통) > 검증(보통) > 나머지(낮음)
- 개발 중:     문제해결(높음) > 검증(높음) > 기획(보통) > 나머지(보통)
- 프로덕션:    검증(높음) > 문제해결(높음) > 회고(높음) > 나머지(보통)
- 유지보수:    문제해결(높음) > 회고(높음) > 검증(보통) > 나머지(낮음)
```

### Step 4: 코딩 도구 매핑 (자동)
```
스택 기반 기본 도구 추천:
- 백엔드 API:  bkit PDCA + g-stack /review + /cso
- 프론트엔드:  bkit PDCA + g-stack /qa + /design-review
- 풀스택:      bkit PDCA + g-stack /review + /qa + /cso
- 데이터/ML:   research-pipeline 중심 + ralph-loop + verification-pipeline
- 인프라/DevOps: g-stack /cso + /ship + verification-pipeline
```

### Step 5: 모델 배분 감사 (v2.5)
```
모든 스킬/에이전트의 model: frontmatter 확인.
판별 기준:
- 전략 판단·아키텍처·보안 감사 → opus
- 코드 구현·반복 루프·QA 실행 → sonnet
- 상태 토글·단순 체크·알림 → haiku
- model: 없이 방치 → 금지 (기본값 무임승차)
```

### Step 6: 시스템 성숙도 트래킹 (v2.5)
```
프로젝트 핏 완료 시 아래 11개 영역 채점 (0~100%).
6개월 목표: 전체 평균 78% 이상.

| 영역 | 현재 | 목표 | 측정 기준 |
|------|------|------|----------|
| 교훈 축적 | — | 80% | CLAUDE.md 교훈 수 ÷ 세션 수 |
| 교훈 승격 | — | 60% | rules/ 파일 수 ÷ 3회+ 반복 교훈 수 |
| 교훈 졸업 | — | 40% | hooks 강제화 수 ÷ rules 위반 반복 수 |
| 모델 배분 | — | 90% | model: 명시 스킬 수 ÷ 전체 스킬 수 |
| 사고-코딩 분리 | — | 95% | 분리 준수 세션 ÷ 전체 세션 |
| 도구 라우팅 | — | 85% | 올바른 도구 선택 ÷ 전체 도구 호출 |
| 등배 체크 | — | 80% | 사고:코딩 비율 3:7~5:5 범위 내 세션 비율 |
| 판단 분류 | — | 70% | 판단/실행 올바른 분류 비율 |
| 복리 지수 | — | 75% | (교훈→규칙→훅 전환율) × 재발방지율 |
| PDCA 완주율 | — | 85% | Check 90%+ 도달 ÷ 전체 PDCA 사이클 |
| 자동 트리거 | — | 70% | 자동 발동 ÷ (자동 + 수동) 트리거 |
```
