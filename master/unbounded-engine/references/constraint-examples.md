# 제약 분류 실전 예시 (Constraint Classification Examples)

> 무한사고엔진 Phase 0(제약 스캔)에서 참조하는 핵심 문서.
> "이 제약이 진짜인가?"를 판별하는 실전 사례 모음.

---

## 판별 프로토콜

```
Q1: "이 우주의 어떤 존재도 이 제약을 깰 수 없는가?"
    → YES: L1 (물리/논리 법칙)
    → NO: Q2로

Q2: "돈/시간/기술이 무한하면 이 제약이 사라지는가?"
    → YES: L2 (기술적 제약)
    → NO: Q3로

Q3: "누가 이 규칙을 만들었고, 왜인가?"
    → 명확한 답이 있으면: L3 (관습적 제약) → 제거 대상
    → 물리법칙에서 유래하면: L1로 재분류
```

---

## L1 — 물리/논리 법칙 (건드릴 수 없는 진짜 제약)

| # | 제약 | 영역 | 왜 L1인가 |
|---|------|------|----------|
| 1 | 빛보다 빠른 정보 전송 불가 | 물리학 | 상대성 이론의 한계 |
| 2 | 열역학 제2법칙 — 엔트로피 감소 불가 | 물리학 | 질서 유지에는 반드시 에너지 투입 필요 |
| 3 | 과거 데이터로 미래를 100% 예측 불가 | 통계학 | 불확실성은 제거할 수 없다 |
| 4 | 동시에 두 곳에 존재할 수 없다 (거시 세계) | 물리학 | 양자 수준에서만 중첩 가능 |
| 5 | 정보를 완벽하게 압축하면 복원 시 손실 | 정보이론 | 섀넌 한계 |
| 6 | 모든 측정은 대상에 영향을 준다 | 물리학/심리학 | 관찰자 효과 |
| 7 | 1+1=2 | 수학 | 공리 체계 |
| 8 | 에너지 보존 — 무에서 유를 만들 수 없다 | 물리학 | 열역학 제1법칙 |
| 9 | 모든 시스템에는 지연(latency)이 존재 | 물리학/공학 | 전파 속도의 물리적 한계 |
| 10 | 복잡계에서는 초기 조건의 작은 차이가 결과를 크게 바꿈 | 수학 | 카오스 이론 |

### L1의 활용법
L1은 "인정하고 그 안에서 최적을 찾는" 대상이다.
"과거 데이터로 100% 예측 불가"라는 L1 안에서,
"100%는 불가능하지만 65%는 가능한가?"가 올바른 질문이다.

## L2 — 기술적 제약 (아직 안 되는 것, 조건이 바뀌면 돌파 가능)

| # | 제약 | 영역 | 왜 L2인가 | 돌파 조건 |
|---|------|------|----------|----------|
| 1 | GPU 메모리 3GB로 대형 모델 불가 | 하드웨어 | 메모리 증설하면 해결 | 신규 GPU 구매 |
| 2 | 바이낸스 API 요청 제한 (1200/분) | 인프라 | API 계정 추가 또는 WS로 전환 | 아키텍처 변경 |
| 3 | QuestDB 단일 서버 처리량 한계 | 인프라 | 샤딩 또는 클라우드 이전 | 비용 증가 |
| 4 | 솔로 개발자 = 동시 작업 불가 | 인력 | 팀 구성 또는 AI 에이전트 활용 | AI 하네스 |
| 5 | 35K봉 백테스트에 Opus 토큰 비용 | 비용 | Stage2(JS 엔진)로 전환 | 개발 시간 투자 |
| 6 | 실시간 처리에 Python이 느림 | 기술 | Rust/Go로 핫패스 재작성 | 개발 시간 |
| 7 | 현재 N-gram 4-gram까지만 검증됨 | 데이터 | 더 많은 데이터 수집+분석 | 시간 |
| 8 | PV2↔U/N 정량 비교 미완료 | 연구 | 동일 구간 비교 실험 수행 | 연구 시간 |
| 9 | 모바일에서 차트 조작 어려움 | UX | 터치 최적화 UI 개발 | 개발 시간 |
| 10 | 3종목 횡전개 미완료 | 검증 | 실험 수행 | 시간+데이터 |

### L2의 활용법
L2는 "조건부 수용"이다.
"비용이 1/10이면?", "시간이 2배 있으면?", "AI가 대신하면?" 질문으로
L2를 L3처럼 제거할 수 있는지 탐색한다.

## L3 — 관습적 제약 (누군가 만든 규칙 → 전부 무시 대상)

| # | 제약 | 영역 | 왜 L3인가 | 무시하면 어떻게 되나 |
|---|------|------|----------|---------------------|
| 1 | "트레이딩 봇은 Python이 표준" | 개발 | 아무도 정한 적 없음. Rust/TS도 가능 | 더 빠른 봇 가능 |
| 2 | "품의서는 PPT로 만든다" | 업무 | 회사 관례일 뿐 | HTML/PDF도 전달 가능 |
| 3 | "백테스트는 수천 라인 코드 필요" | 개발 | JSON 기반 경량 백테스트 가능 | Stage1 Opus 접근 |
| 4 | "AI에게 전략 개발을 맡기면 안 된다" | 편견 | 도구 활용이지 의존이 아님 | 하네스로 품질 보장 |
| 5 | "지표를 많이 쓸수록 정확하다" | 트레이딩 | 제1원리 위반. 원리 없는 지표는 소음 | 소수 핵심 지표 집중 |
| 6 | "스크립트에 피봇 변수는 고정값 사용" | 개발 | TradingView 예시가 그럴 뿐 | PV1/2/3 가변 가능 |
| 7 | "하루 8시간 일하면 충분하다" | 업무 | 시간 ≠ 생산성. 사이클 밀도가 핵심 | 짧은 스프린트 반복 |
| 8 | "알트코인은 BTC보다 위험하다" | 투자 | 전략 의존적. 전략 없으면 BTC도 위험 | 종목별 검증으로 해결 |
| 9 | "트뷰 없이 차트 분석 불가" | 도구 | 인앱 독자 기능이 트뷰를 대체 | P레벨 자동화 등 |
| 10 | "승률 50% 이하는 전략이 아니다" | 트레이딩 | R:R이 높으면 30%로도 수익 | PF 기반 평가 |
| 11 | "회의는 대면이 가장 좋다" | 업무 | 비동기 문서가 더 효율적일 수 있음 | 비동기 우선 |
| 12 | "MVP는 투박해야 한다" | 기획 | 핵심 기능의 완성도가 더 중요 | 작지만 완결된 경험 |
| 13 | "에이전트는 코드만 짠다" | AI | 기획/분석/검증도 가능 | 7분류 체계 |
| 14 | "보고서는 워드로 써야 한다" | 업무 | HTML이 더 인터랙티브할 수 있음 | 시각화+인터랙션 |
| 15 | "한 채팅에서 한 주제만" | AI 활용 | 프로젝트 단위 사고가 더 효율적 | 7분류 오케스트레이션 |

### L3의 활용법
L3을 발견하면 즉시 질문한다:
"이 규칙을 만든 사람이 누구고, 그 이유가 지금도 유효한가?"
유효하지 않으면 → 제거.
제거한 후 문제를 다시 보면 → 대부분 더 단순해지거나 사라진다.

## 혼동하기 쉬운 경계 사례

| 제약 | 얼핏 보이는 분류 | 실제 분류 | 이유 |
|------|----------------|----------|------|
| "GTX 1060으로 LLM 학습 불가" | L1 | **L2** | 클라우드 GPU를 쓰면 해결. 하드웨어 제약은 L2 |
| "과거 패턴이 반복된다고 확신할 수 없다" | L1 | **L1** | 불확실성은 물리적 한계. 하지만 "확률적으로 반복 경향이 있다"는 활용 가능 |
| "차트는 과학이 아니다" | L1 | **L3** | 편견. 차트는 심리의 수학적 결정체(도리님 원리) |
| "코딩 없이 자동매매 불가" | L1 | **L3** | 노코드 플랫폼 존재. 코딩은 수단이지 필수가 아님 |
| "바이낸스가 API를 막으면 끝" | L1 | **L2** | 다른 거래소 API 또는 직접 체결 가능 |
| "하네스 엔지니어링은 대기업용" | L3 | **L3** | 솔로 개발자에게도 적용 가능 (MGTG가 증명 중) |

## Chesterton 체크 필요 사례 (신규 · ICLR 2024 LLM 환각 방어)

> Econlib (2024) "Misusing Chesterton's Fence" — L3 판정 후 제거 전 3질문 체크 필수.
> Q1(역사): 언제/어떤 사건 이후에 생겼나? Q2(고통): 이 규칙이 없었다면 어떤 고통이 발생했나? Q3(환각): 내 설명이 학습 데이터의 그럴듯한 사후 합리화인가?

| # | 제약 | 초기 분류 | Q1 역사 | Q2 고통 | Q3 LLM 환각 체크 | Chesterton 결과 | 최종 분류 |
|---|------|----------|---------|---------|-----------------|----------------|---------|
| 1 | "품의서는 PPT로 만든다" | L3 | 2010년대 사내 회람 관행 | 대면 회의용 시각 보조 편의 | 낮음 (근거 명확) | ✅ **통과** → 제거 가능 | L3 확정 |
| 2 | "거래소 API 요청에는 항상 재시도 로직 포함" | L3 후보 | 2010 Flash Crash, 2021 코인베이스 장애 등 실제 사고 | 미싱 응답 → 포지션 유령화 → 무제한 손실 | 높음 위험 (AI가 "복원력 중요"로 그럴듯 설명) | ❌ **실패** → L2로 재분류 | **L2 기술제약** |
| 3 | "백테스트는 IS/OOS 분리" | L3 후보 | 1960s Markowitz 후 수많은 curve-fit 참사 | 미분리 시 과적합 → 실전 0 실패 확정 | 매우 높음 (AI가 "통계 편의"로 얼버무림) | ❌ **실패** → L1 통계적 필연 | **L1 물리/논리** |
| 4 | "TradingView 스크립트에 피봇은 고정값" | L3 | 2018 TV 샘플 코드 관행 | 없음 (가변 대체 즉시 가능) | 낮음 | ✅ **통과** → 제거 가능 | L3 확정 |
| 5 | "파일명에 공백을 넣지 않는다" | L3 후보 | 1970s Unix shell 쿼팅 실패 축적 | CI/CD·셸 스크립트 깨짐 | 중간 (AI가 "그냥 관행"으로 얼버무림) | ❌ **실패** → L2로 재분류 | **L2 기술제약** |
| 6 | "리스크 관리는 Kelly 공식으로" | L3 후보 | Kelly 1956 정보이론 기원 | 과소 레버리지 시 수익 정체 | 높음 (AI가 "최적 성장"으로 단정) | ❌ **실패** → L1 수학적 상한 | **L1 수학** (상한에서) |
| 7 | "승률 50% 이상이어야 전략" | L3 | 1990s 리테일 트레이딩 교재 통설 | 없음 (PF/RR로 대체 가능) | 낮음 | ✅ **통과** → 제거 가능 | L3 확정 |
| 8 | "테스트는 먼저 작성한다 (TDD)" | L3 후보 | 2000s Kent Beck TDD 운동 | 코드 수정 시 회귀 검출 실패 누적 | 높음 (AI가 "좋은 관행"으로 얼버무림) | ⚠️ **보류** → 맥락 의존 | [UNVERIFIED L3] |

### Chesterton 체크 운영 규칙

- ALWAYS: L3 판정 후 즉시 제거하기 전 3질문 체크 필수
- ALWAYS: Q3 (LLM 환각) 답이 "높음/매우 높음" → 자동으로 [UNVERIFIED L3] 태그
- NEVER: Chesterton 체크 없이 L3 삭제 — 실전 사고 유발 가능 (위 #2, #3, #5, #6 참조)
- NEVER: Chesterton 실패 사례를 "편의상 통과" 처리 — 분류 격상 (L2/L1)이 정직한 방어

### 재분류 효과

| 분류 이동 | 대응 변화 |
|----------|----------|
| L3 → L2 | 제거 대상 → 조건부 수용 (돌파 조건 탐색) |
| L3 → L1 | 제거 대상 → 인정하고 그 안에서 최적화 |
| L3 → [UNVERIFIED] | 제거 대상 → 파일럿 실험으로 검증 후 결정 |

## 제약 제거 후 문제 재정의 패턴

| 원래 문제 | L3 제약 | 제거 후 재정의 |
|----------|---------|--------------|
| "승률 65% 전략을 찾아야 한다" | 승률 기준이 절대적이라는 관습 | "PF 2.0 이상 전략을 찾는다" (승률 무관) |
| "TradingView 스크립트를 완성해야 한다" | 트뷰가 최종 도구라는 관습 | "검증 레이어를 만든다" (인앱이 최종) |
| "전략을 100개 만들어야 한다" | 많을수록 좋다는 관습 | "3개를 완벽하게 검증한다" |
| "모든 종목에 통하는 전략이 필요하다" | 범용성 = 좋다는 관습 | "4종목 중 3/4 유효면 확정" |
| "품의서를 매번 수동으로 만들어야 한다" | 자동화가 안 된다는 관습 | "스킬이 자동 생성한다" |
