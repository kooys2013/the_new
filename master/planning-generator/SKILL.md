# 기획 생성기 (planning-generator) v2.0

> "기획서는 완성품이 아니라 토론의 안건이다. 빠른 80%가 느린 100%를 이긴다."
> — Manyfast Vibe Planning 철학

---

## v2.0 변경 요약

| 항목 | v1.x | v2.0 |
|------|------|------|
| 기획 단계 | PRD → FS → IA (3단계) | PRD → FS → IA → User Flow (4단계) |
| FS 명칭 | 기능명세서 (FS) | 기능명세서 (Features) |
| FS 계층 | 3레벨 자유 서술 | 4레벨, 레벨별 필수 속성 표준화 |
| 핵심 원칙 | 6개 | 8개 (기계 가독성, 연결 추적성 추가) |
| 검증 체크리스트 | 15항목 | 22항목 (연결 검증·기계 가독성 강화) |
| 출력 포맷 | 텍스트/HTML/PPT | + MCP 호환 JSON, Excel, 이미지 |
| 참고 문서 | docs.manyfast.io v1 | docs.manyfast.io 2026.03 최신 |

## 핵심 철학 (반드시 내면화하라)

| 원칙 | 설명 |
|------|------|
| **Why > How** | '어떻게 만들지'보다 '왜 필요한지'를 먼저 정의한다 |
| **80/20 기획** | 목표·타겟·핵심플로우·비즈니스모델의 80%를 먼저, 나머지 20%는 팀과 함께 |
| **속도 > 완벽** | 3일 초안이 3주 완성본을 이긴다. 빠른 초안은 저비용 학습이다 |
| **기계 가독성** | AI 코드 에디터(Cursor, Claude Code)가 파싱 가능한 구조화 문서를 만든다 |
| **연결 추적성** | PRD 역할 → FS Feature → FS Spec → IA 페이지 → User Flow 전체가 양방향 추적 가능 |
| **살아있는 시스템** | PRD·FS·IA·UF 4문서가 상호연결. 하나가 바뀌면 연쇄 영향을 자동 추적 |
| **기획자 = 퍼실리테이터** | 완벽한 설계사가 아니라 집단 지성을 끌어내는 역할 |
| **복합 개선** | 기획에서 빠뜨린 항목은 체크리스트에 영구 추가. 같은 실수 반복 방지 |

## Phase 0: 프로젝트 유형 진단 (Gate & Classify)

요청을 받으면 **반드시 먼저** 아래 분류로 프로젝트 유형을 판별하라.

### 기획 유형 분류표

| 유형 | 키워드 | 주요 산출물 |
|------|--------|-------------|
| **A. 신규 제품/서비스** | 앱, SaaS, 웹서비스, MVP, 랜딩페이지 | PRD + FS + IA + User Flow 전체 |
| **B. 기능 추가/개선** | 기능 추가, 개선, 리뉴얼, 업데이트 | FS 중심 (Feature·Spec) + User Flow |
| **C. 사업/비즈니스 기획** | 사업계획, BM, 수익모델, 시장분석 | PRD 중심 + BM Canvas |
| **D. 마케팅/컨텐츠 기획** | 캠페인, 콘텐츠, 랜딩페이지 카피 | 포지셔닝 + 메시지 구조 |
| **E. 업무/프로세스 기획** | 업무 프로세스, SOP, 운영 기획 | 문제해결 스킬로 위임 권장 |

### 유형별 기본 출력 형식

| 유형 | 기본 형식 | 시각화 포함 여부 |
|------|-----------|-----------------|
| A | PRD→FS→IA→UF 전체 (텍스트) + HTML 시각화 | ✅ 반드시 |
| B | FS 집중 (텍스트) + 기능 계층도 + User Flow | ✅ 권장 |
| C | PRD + BM 캔버스 (텍스트) | ✅ 캔버스 시각화 |
| D | 포지셔닝 프레임 + 카피 구조 | ⚪ 선택 |
| E | → `problem-solver` 스킬로 전환 권장 | — |

## Phase 1: 기획 입력 수집

유형 판별 후, 부족한 정보만 질문하라. **한 번에 최대 3개 질문**.
아래 5필드가 핵심이다 (Manyfast PRD 표준).

### PRD 5핵심 필드

```
1. 개요(Overview)   → 이 기획의 이름/주제 한 줄
2. 목표(Goals)      → 어떤 문제를 해결하고 어떤 가치를 만드는가
3. 타겟 사용자      → 누구를 위한 것인가, 그들의 상황/니즈
4. 사용자 역할      → 시스템 내 역할 유형 (예: 일반사용자, 관리자, 파트너)
                      ※ FS의 Feature에 직접 연결되는 핵심 매핑 키
5. 기기(Devices)    → Web / iOS / Android / Desktop / 내부도구 등
```

### 정보 수집 질문 우선순위

**1순위 (목표·타겟 불명확):**
```
기획을 시작하기 전, 3가지만 확인할게요:

1. 🎯 이 서비스/기능이 해결하는 핵심 문제가 뭔가요?
2. 👥 주요 사용자는 누구이고, 지금 어떤 상황인가요?
3. 📱 어떤 환경에서 사용하나요? (웹/앱/내부도구 등)
```

**2순위 (타겟 있으나 범위 불명확):**
```
1. 이번 MVP/v1에서 반드시 있어야 하는 핵심 기능은?
2. 나중으로 미뤄도 되는 기능은?
```

**즉시 시작 조건:** 개요 + 목표 + 타겟 사용자가 명확하면 바로 기획 생성

### 가정 제한 규칙

```
사용자가 정보를 주지 않아서 가정(Assumption)으로 채우는 경우:
- 가정 3개까지: 명시하고 진행
- 가정 4개 이상: 반드시 사용자에게 확인 질문 (무한 가정 금지)
- 핵심 필드(목표·타겟)의 가정: 1개라도 있으면 즉시 질문
```

### Phase 1-D: 마이크로 태스크 분해 (2-5분 단위)

> **원칙**: 한 태스크 = 한 번의 검증 가능한 산출물.
> 5분 넘게 걸리는 태스크는 **반드시** 더 쪼개야 한다.

#### 측정 가능성 기준

1 마이크로 태스크 = **아래 중 하나**:
- 1 Edit 작업 (파일 1개 + 단일 섹션 수정)
- 1 Read + 1 diff 확인
- 1 Bash 명령 + 출력 검증
- 1 Agent 호출 + 결과 1건 수집

#### 분해 체크리스트

- [ ] 각 태스크의 완료 판정 기준이 **수치 또는 tool call 증거**로 표현되는가?
- [ ] 태스크 간 순서 의존성이 명시되었는가? (병렬 가능 vs 직렬 필요)
- [ ] 모든 태스크가 SCP-5 Verify 단계와 1:1 매핑되는가?
- [ ] 5분 초과 예상 태스크는 재분해되었는가?

#### 분해 예시

```
❌ 나쁜 예: "API 에러 처리 개선"  (측정 불가능)

✅ 좋은 예:
1. (3분) auth.ts의 /login 엔드포인트 에러 타입 추출 (Read + grep)
2. (5분) InvalidCredentials, RateLimited 2종 에러 클래스 추가 (Edit 1회)
3. (4분) unit test 3건 작성 (Edit 1회)
4. (2분) npm test 실행 및 출력 확인 (Bash + SCP-5 Verify)
```

#### 연결

- problem-solver Phase 2-A 가 3회 실패 시 → 여기로 역류 (아키텍처 재분해)
- SCP-5 완료 주장 검증 프로토콜 → 각 마이크로 태스크의 Verify 단계

## Phase 2: PRD (프로덕트 요구사항 문서) 생성

수집된 정보로 PRD를 생성한다.

### PRD 템플릿

```markdown
# PRD: [프로젝트명]

## 1. 개요 (Overview)
[한 문장으로 이 제품이 무엇인지]

## 2. 문제 정의 (Problem Statement)
- 현재 상황: [As-Is]
- 해결할 문제: [Pain Point]
- 해결 후 기대: [To-Be]

## 3. 목표 (Goals)
### 비즈니스 목표
- [정량 목표: 예) 3개월 내 MAU 1,000명]

### 사용자 목표
- [사용자가 얻는 핵심 가치]

### 제외 범위 (Out of Scope)
- [이번 버전에서 하지 않는 것]

## 4. 타겟 사용자 (Target Users)
| 페르소나 | 상황 | 핵심 니즈 | Pain Point |
|---------|------|-----------|------------|
| [주 사용자] | [상황] | [니즈] | [불편함] |
| [부 사용자] | [상황] | [니즈] | [불편함] |

## 5. 사용자 역할 (User Roles) — ※ FS Feature 매핑 키
| 역할 ID | 역할명 | 권한 수준 | 주요 행동 |
|---------|--------|-----------|-----------|
| ROLE-01 | [역할1] | [Admin/User/Viewer] | [핵심 행동] |
| ROLE-02 | [역할2] | [...] | [...] |

## 6. 기기 (Devices & Context)
- 주 환경: [Web / iOS / Android]
- 보조 환경: [있다면]
- 특이 조건: [오프라인 필요 여부 등]

## 7. 성공 지표 (Success Metrics)
| 지표 | 기준값 | 측정 방법 |
|------|--------|-----------|
| [KPI1] | [목표값] | [측정법] |
```

### Why 발견 체크리스트 (PRD 작성 전 반드시 확인)

아래 3가지 소스에서 "왜" 이 기획이 필요한지 근거를 확인하라:

- [ ] **데이터 소스**: 이탈률, 사용 패턴, 전환율 등 수치 근거
- [ ] **현장 소스**: CS팀·팀원에게 "현재 가장 불편한 게 뭔가요?" 확인
- [ ] **사용자 소스**: 직접 사용자 의견 또는 유사 서비스 리뷰 분석

5 Whys 적용: 기획 항목마다 "왜 이게 필요한가?"를 5번 자문하라.

## Phase 3: FS (기능명세서/Features) — 4계층 분해

PRD의 목표를 4단계 계층으로 분해한다.

### 4계층 구조 (v2.0)

```
Requirement (요구사항)           ← 고수준 목표/니즈
  └─ Feature (기능)              ← 요구사항을 충족하는 구체적 기능
       └─ Specification (상세)   ← 기능 내 구체적 동작·정책
            └─ User Flow (흐름)  ← 사용자 시나리오별 실행 경로
```

### FS 작성 규칙 — 레벨별 필수 속성

**Requirement (요구사항) 레벨:**
- 이름, 설명 (필수)
- **Acceptance Criteria**: 체크리스트 형태의 완료 조건 (필수)
- 비즈니스 임팩트 (권장)

**Feature (기능) 레벨:**
- 이름, 설명 (필수)
- **User Role 연결**: PRD에서 정의한 ROLE-ID 매핑 (필수)
- **Importance**: 🔴 High / 🟡 Medium / 🟢 Low (필수)

**Specification (상세기능) 레벨:**
- 이름, 구체적 동작·정책·제약 (필수)
- **IA 페이지 연결**: IA의 특정 페이지 ID와 연결 (필수)

**User Flow (사용자 흐름) 레벨:**
- 시나리오명, 시작→종료 경로 (필수)
- 관련 Spec ID 목록 (필수)
- 정상/예외/에러 경로 구분 (권장)

### FS 템플릿 (Markdown)

```markdown
## FS: [프로젝트명] 기능명세서

### REQ-01: [요구사항명]
**설명**: [이 요구사항이 해결하는 것]

**Acceptance Criteria**:
- [ ] [완료 기준 1]
- [ ] [완료 기준 2]

#### FEAT-01-01: [기능명]
- **User Role**: ROLE-01 ([역할명])
- **Importance**: 🔴 High
- **설명**: [기능 설명]

  ##### SPEC-01-01-01: [상세기능명]
  - [구체적 동작 또는 정책]
  - **연결 페이지**: IA > PAGE-[ID] ([페이지명])

  ##### SPEC-01-01-02: [상세기능명]
  - [구체적 동작 또는 정책]
  - **연결 페이지**: IA > PAGE-[ID] ([페이지명])
```

### 우선순위 결정 프레임워크 (RICE 간소화)

| 기능 | 영향도 (Impact) | 확실성 (Confidence) | 노력 (Effort) | 우선순위 |
|------|----------------|--------------------|--------------|---------| 
| [기능A] | 높음 | 높음 | 낮음 | ⭐ 1순위 |
| [기능B] | 높음 | 중간 | 높음 | 2순위 |
| [기능C] | 낮음 | 낮음 | 높음 | 후순위 |

## Phase 4: IA (정보구조도) — 페이지 계층 설계

기능명세서의 Specification을 실제 화면(페이지)으로 구조화한다.

### IA 설계 원칙

1. **사용자 멘탈 모델 기준**: 개발 편의가 아닌 사용자의 인지 흐름으로 구성
2. **3depth 이내 권장**: 더 깊어지면 사용자 길을 잃는다
3. **네이밍**: 사용자 언어 (관리자 용어 X, 사용자 용어 O)
4. **양방향 연결 추적**: 각 페이지 ↔ FS Spec, 각 Spec ↔ IA 페이지 상호 참조
5. **두 가지 뷰**: 자세히 보기(전체 속성) / 간단히 보기(계층만) 제공

### IA 템플릿

```markdown
## IA: [프로젝트명] 정보구조도

### PAGE-01: 홈 (/) — 1depth
  - 연결 스펙: SPEC-00-01 (메인 대시보드)
  - 역할: 모든 사용자

  ### PAGE-01-01: 기능A (/feature-a) — 2depth
    - 연결 스펙: SPEC-01-01-01
    - 역할: ROLE-01

    ### PAGE-01-01-01: 기능A 상세 (/feature-a/detail) — 3depth
      - 연결 스펙: SPEC-01-01-02
      - 역할: ROLE-01

### PAGE-02: 설정 (/settings) — 1depth
  - 연결 스펙: SPEC-03-01-01
  - 역할: ROLE-01 (Admin)

  ### PAGE-02-01: 계정 설정 (/settings/account) — 2depth
    - 연결 스펙: SPEC-03-01-02
    - 역할: 모든 사용자
```

## Phase 5: User Flow (사용자 흐름) — v2.0 신규

IA 페이지를 사용자의 실제 행동 시나리오로 연결한다.
AI 코드 에디터가 구현 시 참조하는 핵심 실행 경로 문서.

### User Flow 설계 원칙

1. **핵심 시나리오 우선**: MVP에 필요한 Happy Path 먼저
2. **3가지 경로 구분**: 정상(Happy), 예외(Edge), 에러(Error)
3. **FS Spec ↔ IA Page 교차 참조**: 각 단계가 어떤 Spec이고 어떤 Page인지 명시
4. **기계 가독성**: 시나리오를 구조화하여 AI 코드 에디터가 자동 파싱 가능

### User Flow 템플릿

```markdown
## User Flow: [프로젝트명]

### UF-01: [시나리오명] (예: 신규 사용자 회원가입)
- **관련 역할**: ROLE-01
- **시작 조건**: [사용자가 어떤 상태에서 시작하는가]
- **종료 조건**: [성공 시 어떤 상태가 되는가]

**정상 경로 (Happy Path):**
| 단계 | 사용자 행동 | 시스템 응답 | 페이지 | Spec |
|------|-----------|-----------|--------|------|
| 1 | [행동] | [응답] | PAGE-01 | SPEC-01-01-01 |
| 2 | [행동] | [응답] | PAGE-01-01 | SPEC-01-01-02 |
| 3 | [행동] | [응답] | PAGE-01-01-01 | SPEC-01-02-01 |

**예외 경로:**
- [조건] → [어떻게 처리] → [이동할 페이지]

**에러 경로:**
- [에러 상황] → [에러 메시지] → [복구 방법]
```

## Phase 6: 출력 생성 — 상황별 형식 선택

### 출력 형식 결정 로직

```
사용자가 "시각화" / "보기 좋게" / "HTML"     → HTML 인터랙티브 대시보드
사용자가 "PPT" / "발표자료"                  → pumeuiseo-generator 스킬로 전환
사용자가 "간단히" / "빠르게"                  → 텍스트 마크다운
사용자가 "MCP" / "커서용" / "클로드코드용"    → MCP 호환 구조화 JSON
사용자가 "엑셀" / "개발자용"                  → 기획 항목별 Excel 내보내기
사용자가 아무 말 없으면                       → 텍스트 + 선택적 시각화 제안
```

### HTML 기획서 시각화 (show_widget 활용)

HTML 출력 시 아래 섹션을 포함하라:
- **기획 요약 카드**: PRD 5필드 핵심 요약
- **기능 계층도**: Requirement → Feature → Spec → User Flow 트리
- **IA 사이트맵**: 페이지 계층 시각화
- **연결 매트릭스**: PRD 역할 ↔ FS Feature ↔ IA Page 매핑 테이블
- **우선순위 매트릭스**: 영향도 vs 노력 2x2 매트릭스
- **기획 체크리스트**: 완료 여부 표시

### MCP 호환 출력 (v2.0 신규)

AI 코드 에디터에서 바로 소비 가능한 구조화 포맷:

```json
{
  "project": "[프로젝트명]",
  "prd": {
    "overview": "...",
    "goals": { "business": [], "user": [] },
    "target_users": [...],
    "user_roles": [{ "id": "ROLE-01", "name": "...", "permissions": "..." }],
    "devices": [...]
  },
  "fs": {
    "requirements": [{
      "id": "REQ-01",
      "name": "...",
      "acceptance_criteria": [...],
      "features": [{
        "id": "FEAT-01-01",
        "role": "ROLE-01",
        "importance": "high",
        "specs": [{
          "id": "SPEC-01-01-01",
          "page": "PAGE-01-01",
          "description": "..."
        }]
      }]
    }]
  },
  "ia": { "pages": [...] },
  "user_flows": [...]
}
```

### 파일 생성 기준

파일이 필요한 경우 (사용자가 "파일로", "저장", "다운로드" 언급):
- 파일명: `YYMMDDTHHMM_PRD_[프로젝트명].md`
- 경로: 프로젝트 `작업진행/` (Claude.ai: `/mnt/user-data/outputs/`)

## Phase 7: 기획 품질 자가검증 체크리스트

산출물 완성 후 반드시 아래를 확인하라:

### PRD 검증 (5항목)
- [ ] 개요가 한 문장으로 명확한가?
- [ ] 목표가 측정 가능한가? (정량 지표 포함)
- [ ] 타겟 사용자가 구체적인가? (막연한 "20-30대" X)
- [ ] Out of Scope가 명시되었는가?
- [ ] "왜 필요한가"의 근거가 있는가?

### FS 검증 (6항목)
- [ ] 모든 Requirement에 Acceptance Criteria(체크리스트형)가 있는가?
- [ ] 모든 Feature에 User Role ID와 Importance가 있는가?
- [ ] PRD의 모든 ROLE이 최소 1개 Feature에 매핑되었는가?
- [ ] Specification이 구현 가능한 수준으로 구체적인가?
- [ ] 모든 Specification에 IA 페이지 연결이 있는가?
- [ ] 우선순위가 설정되었는가?

### IA 검증 (4항목)
- [ ] 3depth 이내로 구성되었는가?
- [ ] 모든 핵심 페이지가 FS Spec과 양방향 연결되었는가?
- [ ] 사용자 언어로 네이밍되었는가?
- [ ] 각 페이지에 접근 가능한 역할이 명시되었는가?

### User Flow 검증 (4항목)
- [ ] 핵심 시나리오(Happy Path)가 최소 1개 정의되었는가?
- [ ] 각 단계에 FS Spec ID와 IA Page ID가 모두 연결되었는가?
- [ ] 예외/에러 경로가 최소 1개 정의되었는가?
- [ ] 시작 조건과 종료 조건이 명확한가?

### 연결 추적성 검증 (3항목)
- [ ] PRD 역할 → FS Feature → FS Spec → IA Page → User Flow 전체 경로가 끊김 없이 추적 가능한가?
- [ ] 단일 항목 변경 시 연쇄 영향 범위를 특정할 수 있는가?
- [ ] AI에게 "기능 목록 찾아줘"라고 물었을 때 정확한 답이 나올 구조인가? (기계 가독성 테스트)

### 복합 개선 체크
- [ ] 이전 기획에서 빠뜨렸던 항목이 이번에 반영되었는가?
- [ ] 새로 발견된 누락 패턴이 있다면 이 체크리스트에 영구 추가했는가?

## 특수 상황 처리

### 아이디어만 있고 정보가 부족한 경우
가정(Assumption)을 명시하고 기획을 진행한다:
```
⚠️ 가정(Assumption): [가정한 내용]
→ 이 가정이 맞는지 팀과 확인 후 업데이트 필요
```
**제한**: 가정 3개 초과 시 반드시 사용자에게 확인 질문.

### 기존 기획서 개선 요청
1. 기존 문서에서 PRD·FS·IA·UF 4단계 구조 추출
2. 누락된 항목 보완 (특히 User Flow, 연결 추적성)
3. 상호연결 관계 정립
4. 기계 가독성 수준 검증

### MVP vs 풀 제품 기획
MVP 판별 기준:
- 핵심 가정(Core Assumption) 1개를 검증할 수 있는 최소 기능셋
- 사용자가 "아, 이거구나" 경험할 수 있는 수준
- FS에서 🔴 High Importance만 포함
- User Flow에서 Happy Path 1~2개만 정의

### 대형 기획 세션 분리

유형 A(신규 제품) 기획 시 컨텍스트 60% 초과 위험:
- PRD → (확인) → FS → (확인) → IA → (확인) → User Flow 순서로 단계별 확인
- 한 세션에 4단계 전부 강행하지 말 것
- 각 단계 완료 시 사용자 확인 후 다음 진행

## DON'T
- "전체적으로 개선해줘" 같은 모호한 범위 — 탐색 폭발, 불필요 파일 포함
- 한 번에 10개+ 파일 변경 계획 — Phase 없이 검증 불가
- 이전 대화 계획 재사용 — 코드 변경 가능성, 재탐색 필수
- 계획 중 코드 수정 (Edit/Write 호출) — 절대 금지. 읽기 전용

## DO
- 구체적 파일/함수명을 프롬프트에 포함 — 탐색 범위 축소
- "기존 [X] 패턴과 동일하게" — 코드 일관성 앵커
- 접근법 2~3개 비교 후 선택 — 비교 없이 1안만 제시 금지
