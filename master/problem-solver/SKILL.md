# 문제해결기 (problem-solver) v2.0

> "좋은 기획자는 'How'에 집착하지 않는다. 'Why'에 집착한다."
> — Manyfast 기획 철학

---

## v2.0 변경 요약

| 항목 | v1.x | v2.0 |
|------|------|------|
| 부모 스킬 연동 | PRD → FS → IA (3단계) | PRD → FS → IA → User Flow (4단계) |
| 액션 플랜 구조 | ACT-ID 자유 서술 | ROLE-ID 연결 + Importance + 연결 추적성 |
| 해결 흐름 문서 | 없음 | Fix Flow (User Flow 형식의 해결 시나리오) |
| 기술 문제 검증 | 회귀 검증 | + 기계 가독성 검증, 연결 영향 분석 |
| 기획 전환 시 | PRD + FS 전달 | PRD + FS + IA + User Flow 전달 |
| 검증 체크리스트 | 기본 | 연결 추적성 + 기계 가독성 포함 |

---

## 부모 스킬과의 관계

```
planning-generator (부모)              problem-solver (자식)
──────────────────────────            ──────────────────────────
이상적인 미래를 설계한다               현재 문제를 해결한다
PRD → FS → IA → UF로 구조화          Problem → Root Cause → Solution → Fix Flow
신규 서비스·기능 기획                  기존 상황 개선·트러블슈팅
"무엇을 만들 것인가"                  "무엇이 문제이고 어떻게 고칠 것인가"
기계 가독성: 기획 문서 구조화          기계 가독성: 해결책 실행 경로 구조화
```

**연동 패턴:**
- 문제 분석 → 해결책이 새 기능이면 → `planning-generator`로 FS + User Flow 작성
- 기획 중 리스크 발견 → `problem-solver`로 리스크 분석
- 대형 문제: `problem-solver`로 원인 파악 → `planning-generator`로 개선 기획 (4단계 전체)

**공유 자산 (v2.0):**
- ROLE-ID 체계: PRD에서 정의한 사용자 역할 ID를 액션 플랜에서도 동일하게 사용
- 연결 추적성: 문제 → 원인 → 해결 ACT → 영향받는 FS Spec → IA Page 연쇄 추적
- Acceptance Criteria: 부모 스킬과 동일한 체크리스트 형태 완료 조건

## Phase 0: 문제 유형 분류 (Gate & Classify)

요청을 받으면 **먼저** 문제 유형을 판별하라.

### 문제 유형 분류표

| 유형 | 키워드/상황 | 주요 분석 방법 | 산출물 |
|------|------------|---------------|--------|
| **A. 기술·개발 문제** | 버그, 에러, 코드, 빌드 실패, 성능 | 로그 분석, 재현 절차, 코드 리뷰 | 버그 리포트 + 수정 계획 + Fix Flow |
| **B. 업무·프로세스 문제** | 비효율, 병목, 반복 오류, SOP 위반 | 프로세스 맵, 병목 분석 | 개선 액션 플랜 |
| **C. 설비·환경 문제** | 장비 고장, 이상, 설비 트러블 | 5 Whys, FMEA, 점검 로그 | 조치 보고 + 재발 방지 |
| **D. 의사결정·비즈니스** | 선택 갈등, 전략 판단, 옵션 비교 | 의사결정 매트릭스, 리스크 분석 | 의사결정 보고서 |
| **E. 개인·팀 생산성** | 시간 낭비, 집중도, 업무 과부하 | 시간 분석, 우선순위 재정렬 | 개선 일정 |
| **F. 기획·스펙 문제** | v2.0 — 기획 문서 간 불일치, 스펙 누락, 연결 끊김 | GAP 분석, 연결 추적 | 기획 보완 리포트 |

### 복잡도 판별 (분석 깊이 결정)

| 복잡도 | 기준 | 사용할 Phase |
|--------|------|-------------|
| 🟢 단순 | 원인이 명확, 해결책이 자명 | Phase 1 → Phase 4 (빠르게) |
| 🟡 중간 | 원인이 2-3가지 가능성, 검증 필요 | Phase 1 → 5 전체 |
| 🔴 복잡 | 원인 불명, 연쇄 문제, 시스템 레벨 | Phase 1 → 6 전체 + 참조파일 |

## Phase 1: 문제 정의 — "왜?"를 찾는다

### 1-1. 문제 현황 파악 (AS-IS 명확화)

수집해야 할 정보 (모르면 질문, 한 번에 최대 3개):

```
1. 현상(Symptom): 지금 어떤 상태인가? 정확히 무엇이 발생하고 있는가?
2. 발생 시점: 언제부터? 어떤 조건에서 발생하는가?
3. 영향 범위: 얼마나 자주? 몇 명/몇 건에 영향을 미치는가?
4. 기대 상태(TO-BE): 정상이라면 어떤 상태여야 하는가?
5. 이미 시도한 것: 지금까지 어떤 해결을 시도했는가?
```

### 1-2. 문제 정의서 (Problem Statement)

```markdown
## 문제 정의서

**현상 (Symptom)**:
[구체적으로 무슨 일이 벌어지고 있는가]

**발생 조건**:
- 발생 시점: [언제, 어떤 상황에서]
- 발생 빈도: [항상/간헐적/특정 조건]
- 영향 범위: [몇 명/몇 건/어느 시스템]

**기대 상태 (Expected)**:
[정상 작동 시 어떤 상태여야 하는가]

**현재 상태 (Actual)**:
[현재 실제 상태]

**비즈니스 임팩트**:
[이 문제가 방치되면 어떤 손해가 발생하는가]

**연결 영향 (v2.0)**: — 기존 기획 문서가 있는 경우
- 영향받는 FS Spec: [SPEC-ID 목록]
- 영향받는 IA Page: [PAGE-ID 목록]
- 영향받는 User Flow: [UF-ID 목록]

**해결 우선순위**: 🔴 긴급 / 🟡 보통 / 🟢 여유
```

## Phase 2: 근본 원인 분석 (Root Cause Analysis)

### 2-1. 5 Whys 기법 (기본 — 모든 유형에 적용)

"왜?"를 5번 반복하여 증상이 아닌 근본 원인에 도달한다.

**5 Whys 실행 규칙:**
- 각 "왜?"는 직전 답변에서 파생되어야 한다
- 데이터·증거로 뒷받침할 수 없는 답변은 "가정"으로 표시
- 5번 전에 근본 원인에 도달하면 중단 가능

```markdown
## 5 Whys 분석

**현상**: [문제 현상]

Why 1: 왜 [현상]이 발생했는가?
→ [답변 1] (근거: [데이터/로그/관찰])

Why 2: 왜 [답변 1]이 발생했는가?
→ [답변 2] (근거: [데이터/로그/관찰])

Why 3: 왜 [답변 2]가 발생했는가?
→ [답변 3] (근거: [데이터/로그/관찰])

Why 4: 왜 [답변 3]이 발생했는가?
→ [답변 4] (근거: [데이터/로그/관찰])

Why 5: 왜 [답변 4]가 발생했는가?
→ **근본 원인**: [최종 답변] ← 이것을 해결해야 한다
```

### 분석 제한 규칙 (무한 루프 방지)

```
5 Whys 1차 → 근본 원인 미도달 → 가설 기반 2차 시도
5 Whys 2차 → 여전히 미특정 →
  ⚠️ 에스컬레이션:
  "현재까지 분석 결과를 정리합니다. 근본 원인 특정을 위해
   추가 데이터/로그/현장 확인이 필요합니다."
  → 분석 결과 + 후보 원인 목록 + 필요 데이터를 사용자에게 전달
  → 무한 분석 루프 금지
```

### Phase 2-A: 체계적 디버깅 프로토콜

> **철칙 (blocking)**: 근본원인 미확인 시 수정 진행 **금지**.
> "빠른 임시방편", "이것 한번 시도" 같은 표현 감지 시 즉시 Phase 2-A Step 1로 강제 복귀.

#### 4단계 프로토콜

```
Step 1. 조사 (Investigate)
  - 에러 메시지 전문 읽기 (요약 금지)
  - 재현 일관성 확인 (100% / 간헐적 / 1회성)
  - 최근 변경사항 review (git log -p -5)
  - 계층별 입출력 로깅 (데이터 흐름 역추적)

Step 2. 패턴 분석 (Pattern)
  - 정상 동작하는 유사 코드 찾기
  - 참고 구현과 전체 비교 (diff)
  - 정상 코드와의 차이점 전부 나열
  - 필요한 의존성/설정 파악

Step 3. 가설 수립 및 검증 (Hypothesis)
  - 구체적 가설: "X가 근본 원인이다. 이유는 Y"
  - 최소 변경으로 1개 변수만 테스트
  - 성공 → Step 4
  - 실패 → 새 가설 (Step 3 반복)

Step 4. 구현 (Implement)
  - 재현 테스트 케이스 작성 먼저
  - 근본 원인만 수정 (증상 아님)
  - 수정 후 검증 (service-completion-checklist 5단계 프로토콜 호출)
```

#### 실패 에스컬레이션

| 실패 횟수 | 조치 |
|-----------|------|
| 1~2회 | Phase 2-A Step 3 가설 재수립 |
| **3회 이상** | 아키텍처 재검토 필요 → **planning-generator Phase 1-D 마이크로 태스크 재분해** + unbounded-engine 재진입 |

#### 위험 신호 (즉시 중단 → Step 1 복귀)

- "빠른 임시방편으로 해보자"
- "이것저것 시도해보자" (여러 수정 동시 적용)
- "충분히 이해 못 했지만 이렇게 해보자"
- 근본 원인 대신 증상만 가리는 수정

#### Hotfix 예외 플래그

진짜 프로덕션 장애(서비스 다운, 사용자 영향 진행 중)만 허용:
- 사용자가 명시적으로 "hotfix" 또는 "응급" 선언
- 수정 후 **24시간 이내** Phase 2-A 전체 수행하여 근본원인 기록 필수
- accumulated-lessons에 `[HOTFIX→RCA]` 태그로 기록

### 2-4. 기획·스펙 GAP 분석 (v2.0 신규 — 유형 F)

기획 문서 간 불일치·연결 끊김을 분석:

```markdown
## 기획 GAP 분석

### 연결 끊김 진단
| 소스 | 타겟 | 상태 | 비고 |
|------|------|------|------|
| ROLE-01 | FEAT-XX | ❌ 미연결 | Feature에 역할 누락 |
| SPEC-01-01 | PAGE-XX | ❌ 미연결 | IA 페이지 미생성 |
| UF-01 Step 3 | SPEC-XX | ⚠️ 불일치 | Spec 설명과 Flow 행동 상이 |

### 누락 항목
- [ ] [누락된 FS Spec]
- [ ] [누락된 IA Page]
- [ ] [누락된 User Flow 시나리오]
```

## Phase 3: 해결책 설계

### 3-1. 해결책 3계층 구분

```
┌─────────────────────────────────────────┐
│  임시 조치 (Workaround)                  │
│  → 당장의 증상을 멈추는 응급 처치        │
│  → 빠르게 실행 가능, 근본 원인 미해결    │
├─────────────────────────────────────────┤
│  단기 해결 (Short-term Fix)              │
│  → 직접적 원인을 제거하는 수정           │
│  → 며칠~수 주 내 실행 가능              │
├─────────────────────────────────────────┤
│  영구 해결 (Long-term Solution)          │
│  → 근본 원인을 제거하는 구조적 변경      │
│  → 재발 방지 포함, 기획 필요할 수 있음  │
│  → planning-generator 4단계 연동 대상   │
└─────────────────────────────────────────┘
```

### 3-2. 해결책 후보 평가 매트릭스

```markdown
## 해결책 평가

| 해결책 | 효과 (1-5) | 실현가능성 (1-5) | 비용 (1-5) | 속도 (1-5) | 총점 | 추천 |
|--------|-----------|----------------|-----------|-----------|------|------|
| [방안1] | 5 | 3 | 4 | 2 | 14 | ⭐ |
| [방안2] | 3 | 5 | 5 | 5 | 18 | ⭐⭐ |
| [방안3] | 5 | 2 | 2 | 1 | 10 | |

**추천 방안**: [방안2]
**이유**: [선택 근거]
```

해결책이 명확한 경우, 부모 스킬의 FS 구조로 액션을 정의한다:

```markdown

- **Importance**: 🔴 High
- **기한**: [오늘/내일/이번 주]
- **Acceptance Criteria**:
  - [ ] [완료 확인 기준 1]
  - [ ] [완료 확인 기준 2]

  #### 세부 액션
  - ACT-01-01: [구체적 행동 1]

  - ACT-01-02: [구체적 행동 2]

- **담당 역할**: ROLE-[ID]
- **Importance**: 🟡 Medium
- **기한**: [날짜]
```

```markdown
| 3 | [행동] | [확인 방법] | ROLE-02 | ACT-02 |
```

## Phase 4: 실행 계획 + 검증

### 4-1. 실행 우선순위 결정

```markdown
## 실행 로드맵

**즉시 (오늘~내일)**:
- [ ] [임시 조치] — 담당: ROLE-[ID], 기한: [날짜]

**단기 (이번 주~다음 주)**:
- [ ] [직접 원인 제거] — 담당: ROLE-[ID], 기한: [날짜]

**중기 (이번 달)**:
- [ ] [재발 방지 구조 개선] — 담당: ROLE-[ID], 기한: [날짜]
- [ ] [모니터링 설정] — 담당: ROLE-[ID], 기한: [날짜]
```

### 4-2. 검증 기준 (Verification Criteria)

```markdown
## 해결 검증 기준

**임시 조치 완료 확인**:
- [ ] [증상이 더 이상 발생하지 않는지 확인 방법]

**근본 원인 제거 확인**:
- [ ] [재현 절차로 더 이상 재현되지 않음]
- [ ] [영향 받은 지표가 정상 범위로 돌아옴]

**재발 방지 확인**:
- [ ] [모니터링 알림 설정 완료]
- [ ] [관련 문서/SOP 업데이트 완료]
- [ ] [팀 공유/교육 완료]

**연결 영향 확인 (v2.0)**:
- [ ] [수정으로 인해 기존 FS Spec이 깨지지 않았는가]
- [ ] [수정으로 인해 기존 User Flow가 변경되었다면 UF 문서 업데이트]
- [ ] [수정 후 IA 페이지 구조에 변경이 필요한가]
```

### 기술 문제 회귀 검증 의무

```
기술·개발 문제(유형 A) 해결 후 반드시:
1. 수정한 코드의 컴파일/타입체크/린트 통과 확인
2. 관련 테스트 케이스 재실행
3. 해결로 인해 다른 기능이 깨지지 않았는지 확인
4. (v2.0) 영향받는 FS Spec → IA Page → User Flow 연쇄 변경 확인
```

### 해결 효과 모니터링 게이트

```
영구 해결(Long-term Solution) 적용 후:
- 1주 내 재발 여부 모니터링 기준 명시
- 재발 시: problem-solver 재진입 + 복합 엔지니어링 적용
  (= 같은 문제가 다시 발생하지 않도록 체크리스트·규칙·hook에 영구 추가)
- 미재발 확인: 해결 완료 처리
```

## Phase 6: 구조적 개선 — planning-generator 연동

문제 해결 후 재발을 막기 위해 **구조적 변경이 필요한 경우**:

```
이 문제가 반복된다면 → planning-generator로 개선 기획 수립 (4단계 전체)
이 문제가 시스템 레벨 개선을 요구한다면 → PRD + FS + IA + User Flow 작성 필요
```

**연동 트리거 조건:**
- 같은 문제가 2번 이상 발생
- 임시 조치만으로는 근본 해결 불가
- 팀 프로세스·시스템 변경이 필요
- (v2.0) 기존 기획 문서의 구조적 결함이 문제의 원인인 경우

**전환 시 건네줄 정보 (v2.0):**
```
planning-generator에 전달:
- 개요: "[문제 반복 방지를 위한 개선 기획]"
- 목표: "[재발 방지 및 운영 효율화]"
- 현재 문제: [이번 problem-solver 분석 결과]
- 타겟: [영향받는 팀/사용자]
- 사용자 역할: [기존 ROLE-ID 체계 계승]
- 영향받는 기존 항목:
  - FS Spec: [SPEC-ID 목록]
  - IA Page: [PAGE-ID 목록]
  - User Flow: [UF-ID 목록 + 어떤 경로가 깨졌는지]
```

## 특수 상황 처리

### 긴급 상황 (P0 — 즉시 대응)

프로덕션 장애, 데이터 손실, 보안 사고 등:

```markdown
🚨 긴급 대응 프로토콜

1. [30초] 문제 범위 확인: 얼마나 많은 사용자/시스템에 영향?
2. [5분]  임시 조치 실행: 피해 확산 차단 (롤백/차단/격리)
3. [즉시] 관련자 공유: Slack·이메일로 현황 공유
4. [진행] 근본 원인 분석: 5 Whys 시작
5. [완료] 영구 해결 + 사후 검토(Post-mortem) + 연결 영향 반영
```

### 원인 불명 상황

5 Whys를 해도 원인을 특정할 수 없는 경우:

```markdown
가설 기반 접근법:
1. 가능성 있는 원인 후보 모두 나열
2. 각 가설을 반증할 수 있는 테스트 설계
3. 가장 빠르게 검증 가능한 가설부터 순차 검증
4. 반증된 가설 제거 → 남은 가설 = 원인

[빠른 검증 → ralph-loop 스킬 활용 가능]

제한: 가설 검증 2회 반복 후에도 미특정 → 에스컬레이션
```

### 의사결정 갈등 (옵션 A vs B)

무엇을 선택할지 고민되는 경우:

```markdown
## 의사결정 프레임워크

|-----------|--------|---------|---------|
| 비용 | 30% | 4 | 2 |
| 효과 | 40% | 3 | 5 |
| 속도 | 30% | 5 | 3 |
| **가중 합산** | | **3.8** | **3.4** |

**추천**: [A]
**단, 이런 경우엔 B 선택**: [조건]
```

## DON'T
- 추정으로 수정 시도 — 원인 미확인 상태 코드 변경 = 2차 버그
- 3회 연속 e2e 실패 후 계속 — 사용자 보고 + Rollback 여부 확인
- 근본 원인 없이 핫픽스만 — 같은 버그 반복
- "아마 이게 원인" — 로그 추가해서 실증 확인

## DO
- 디버그 순서: 로그 먼저 → 스크린샷 → 원인 1개 특정 → 수정 1회
- grep으로 동일 패턴 전수 조사 — 다른 곳에도 같은 버그 확률 높음
- 해결 후 ALWAYS/NEVER 규칙 추가 (복리 개선)
