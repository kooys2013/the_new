# 검증 파이프라인 (verification-pipeline)

> "테스트를 통과했다고 안전한 것이 아니다.
>  프로덕션 장애를 출시 전에 상상할 수 있는 자만이 안전하다."
> — gstack Regression Iron Rule + Harness QA 철학

---

## 4형제 파이프라인에서의 위치 — 횡단 레이어

```
┌─────────────────────────────────────────────────────┐
│                 verification-pipeline                │ ← 횡단(cross-cutting)
│  Pre-flight │ In-flight │ Post-flight │ Go/No-Go    │    모든 단계에 삽입 가능
├─────────────┼───────────┼─────────────┼─────────────┤
│  planning   │  research │  problem    │  (최종)      │ ← 순차 파이프라인
│  -generator │  -pipeline│  -solver    │  검증 보고서  │
└─────────────┴───────────┴─────────────┴─────────────┘
```

**횡단 레이어란:** 검증은 마지막 단계에만 있는 게 아니다.
- 기획 시작 전: Pre-flight (요구사항 완전성)
- 연구 중: In-flight (소스 신뢰도, 중간 결과 일관성)
- 문제해결 후: Post-flight (해결책 효과, 재발 여부)
- 전체 완료 후: 최종 Go/No-Go 게이트

---

## 핵심 원칙 (반드시 내면화하라)

| 원칙 | 설명 |
|------|------|
| **현실과 문서의 일치(Field-to-Doc Match)** | P&ID가 실제 배관과 다르면 안 되듯, 모든 주장은 현실과 일치해야 한다 |
| **적응적 리고르(Adaptive Rigor)** | 일상 질문에 DO-178C를 적용하지 마라. 위험도에 비례하는 검증 |
| **적대적 검증(Adversarial Challenge)** | 확인 편향을 깨라. "이것이 틀릴 수 있는 이유"를 적극 탐색 |
| **3연속 실패 = 자동 중단** | 카나리 배포든, 트레이딩이든, 3회 연속 기준 위반 시 롤백/중단 |
| **수치화된 게이트** | "느낌상 괜찮음"은 허용하지 않는다. 0~100 점수 + Pass/Fail 임계값 |
| **메타 검증** | 검증 프로세스 자체도 검증한다. 탈출률(Escape Rate) 추적 |
| **Fix-First** | 명백한 오류는 보고만 하지 말고 즉시 수정. 판단이 필요한 것만 인간에게 |

## Phase 0: 검증 대상 분류 (Gate & Classify)

### 검증 대상 유형

| 유형 | 트리거 | 주요 검증 방법 | 산출물 |
|------|--------|--------------|--------|
| **A. 기획 검증** | PRD, FS, IA, 요구사항 | 완전성·일관성·실현가능성 점검 | 기획 품질 카드 |
| **B. 연구 검증** | 근거 테이블, 리서치 브리프 | 출처 확인·등급 재평가·편향 점검 | 근거 신뢰도 보고서 |
| **C. 해결책 검증** | 수정안, 액션 플랜, 코드 변경 | 근본원인 대응 여부·부작용 점검 | 해결 유효성 보고서 |
| **D. 소프트웨어 검증** | 코드, 배포, 인프라 변경 | 테스트·보안·성능·카나리 | 배포 승인 체크리스트 |
| **E. 안전/법규 검증** | PSM, 설비, 환경 허가 | 법적 요건 대조·현장 기준 확인 | 컴플라이언스 보고서 |
| **F. 트레이딩 검증** | 전략, 백테스트, 파라미터 | WFA·몬테카를로·페이퍼 트레이딩 | 전략 승인 게이트 보고서 |

### 리고르 레벨 결정 (3-Question Triage)

```
Q1. 실패 시 물리적 위험, 1% 이상 재무 손실, 또는 법적 위반 가능성?
    → Yes: L3 이상 (최소)
    
Q2. 1시간 내 원복 가능하고 영구적 영향 없음?
    → Yes: 1단계 하향 가능
    
Q3. 도메인에 법적 강제 기준이 존재? (PSM, ISO, 금융규제 등)
    → Yes: 해당 규제가 하한선
```

| 리고르 | 적용 상황 | 검증 활동 | gstack 대응 |
|--------|----------|----------|-------------|
| **L0 — 최소** | 일상 메모, 내부 노트 | 셀프 체크, 맞춤법 | — |
| **L1 — 표준** | 일반 업무, 일상 조사 | 자동 린트 + 기본 교차 확인 | Tier 1 (bun test) |
| **L2 — 강화** | API 변경, 발표 자료, 재무 분석 | 다중 소스 교차 검증 + 2인 리뷰 | Tier 2 (test:gate) |
| **L3 — 엄격** | PSM, 라이브 트레이딩, 규제 제출 | 독립 검증 + 공식 서명 + 카나리 | Tier 3 (test:evals) |
| **L4 — 완전** | 생명 안전, 항공, 재난 위험 | 공식 V&V + 제3자 인증 | DO-178C DAL A |

## Phase 1: Pre-flight 검증 (실행 전)

### 범용 Pre-flight 체크리스트

| # | 항목 | 확인 내용 | Pass 기준 |
|---|------|----------|----------|
| 1 | **입력 검증** | 필요한 데이터/자원이 모두 존재하는가 | 필수 입력 100% 확보 |
| 2 | **형식 검증** | 입력이 예상 스키마/포맷에 맞는가 | 파싱 에러 0건 |
| 3 | **환경 준비** | 실행 환경 전제조건 충족 | 의존성 전부 가용 |
| 4 | **성공 기준** | 측정 가능한 성공 기준이 정의되었는가 | 최소 1개 정량 기준 존재 |
| 5 | **롤백 기준** | 실패 시 되돌릴 수 있는가 | 롤백 절차 문서화 |
| 6 | **리스크 평가** | 알려진 위험이 식별·완화되었는가 | 고위험 항목 0건 미대응 |

### 스킬별 Pre-flight 확장

| 대상 스킬 | 추가 확인 |
|-----------|----------|
| **planning-generator** | FINER 점수 ≥7/10, 이해관계자 식별 완료, 일정 내부 모순 없음 |
| **research-pipeline** | 연구 질문 PICO 분해 완료, 최소 2개 DB 검색 전략, 포함/배제 기준 사전 정의 |
| **problem-solver** | 문제 진술 정량화, 조사 범위 한정, 롤백 계획 존재 |

### Pre-flight 판정

- **PASS**: 모든 필수 항목 충족 → 진행
- **FAIL**: 1개 이상 필수 항목 미충족 → 차단 + 시정 요구
- **WARN**: 비필수 항목 미충족 → 리스크 수용 문서화 후 진행 (2인 승인)

## Phase 2: In-flight 검증 (실행 중)

### 모니터링 규칙

| 도메인 | 모니터링 지표 | 임계값 | 위반 시 조치 |
|--------|-------------|--------|-------------|
| **소프트웨어** | 에러율, P50 레이턴시, 콘솔 에러 | 에러율 ≤1%, P50 ≤ 기준+20% | 3연속 위반 → 자동 롤백 |
| **연구** | 소스 간 일치율, 중간 합성 일관성 | ≥3개 소스 일치 = 높은 확신 | 불일치 발견 → 추가 조사 |
| **트레이딩** | P&L vs 몬테카를로 밴드 | 5th percentile 20연속 하회 금지 | 위반 → 전략 일시 중지 |
| **안전/법규** | 현장-문서 일치율 | 100% 일치 필수 | 불일치 → 즉시 시정 |

### 중간 체크포인트 (L2 이상)

```
매 주요 단계 완료 시:
  ✅ 이전 단계 산출물과 현재 단계 입력의 정합성 확인
  ✅ 스코프 드리프트(범위 이탈) 감지 — gstack /review 패턴
  ✅ 누적 소요 시간/토큰 vs 예산 확인
```

## Phase 3: Post-flight 검증 (실행 후)

### 범용 Post-flight 체크리스트

| # | 항목 | 확인 내용 | Pass 기준 |
|---|------|----------|----------|
| 1 | **기능 검증** | 산출물이 성공 기준을 충족하는가 | 모든 수락 기준 통과 |
| 2 | **부작용 점검** | 의도하지 않은 영향이 없는가 | 신규 이슈 0건 (또는 문서화) |
| 3 | **데이터 무결성** | 데이터가 손상·누락 없이 완전한가 | 완전성 100% |
| 4 | **추적성 확인** | 모든 주장 → 출처 역추적 가능한가 | 미추적 주장 0건 |
| 5 | **문서화** | 결과와 과정이 문서에 반영되었는가 | 문서-산출물 일치 |
| 6 | **베이스라인** | 새로운 성능 기준이 기록되었는가 | Before/After 비교 존재 |

## Phase 4: 도메인별 검증 체크리스트

### D-1. 소프트웨어 배포 검증 (gstack 참조)

gstack의 14단계 파이프라인에서 추출한 핵심 게이트:

| 단계 | gstack 스킬 | 검증 내용 | Pass 기준 |
|------|------------|----------|----------|
| 계획 | /plan-eng-review | 아키텍처 다이어그램 + 테스트 매트릭스 | 커버리지 ≥70% |
| 코드 | /review | 2-pass 구조 감사 + 스코프 드리프트 탐지 | 크리티컬 이슈 0건 |
| 보안 | /cso | OWASP Top 10 + STRIDE 위협 모델 | 신뢰도 8/10 이상만 보고 |
| QA | /qa | 실 브라우저 Chromium 테스트 | 건강 점수 ≥80/100 |
| 배포 | /ship + /land-and-deploy | CI 통과 + 프로덕션 확인 | 모든 CI 체크 그린 |
| 카나리 | /canary | 배포 후 모니터링 루프 | 콘솔 에러 0, 성능 회귀 없음 |
| 벤치마크 | /benchmark | Core Web Vitals Before/After | LCP/FID/CLS 악화 없음 |

### D-2. 산업안전/환경 검증 (KOSHA/PSM)

| 항목 | 점검 기준 | Pass 기준 | 법적 근거 |
|------|----------|----------|----------|
| **PSM 이행상태** | 12요소 점검 | P(적합) 또는 S(양호) | 산안법 제44조 |
| **위험성평가** | 5단계 절차 이행 | 작업자 참여 + 문서화 완료 | 산안법 시행규칙 |
| **설비 안전검사** | 벽두께, 정격하중, 안전밸브 | 법정 기준 이상 | KOSHA 유해위험기계 |
| **대기 배출** | 허가 배출량 vs 실측값 | 허가 기준 이내 | 대기환경보전법 |
| **변경관리(MOC)** | 변경 전 위험성평가 완료 | 승인 서명 + 교육 완료 | PSM 12요소 |
| **FAT/SAT** | 공장수락시험 → 현장수락시험 | Cat.A 펀치 0건 → 출하 | IQ/OQ/PQ |

### D-3. 트레이딩 전략 검증 (MGTG 게이트)

| 게이트 | 검증 방법 | Pass 기준 |
|--------|----------|----------|
| **Gate 1 — 백테스트** | IS 기간 수익성 | 양의 기대값 + 현실적 수수료 반영 |
| **Gate 2 — WFA** | Walk-Forward Efficiency | WFE ≥50%, OOS 70%+ 윈도우 수익성 |
| **Gate 3 — 몬테카를로** | 1,000+ 시뮬레이션 | 중앙값 수익 양(+), 파산확률 <5% |
| **Gate 4 — 페이퍼** | 30~90일 실시간 모의 | 몬테카를로 밴드 내 성과 |
| **Gate 5 — 라이브** | 10~20% 축소 배분 1~3개월 | 샤프 백테스트 대비 50% 이내 |

**킬 스위치:** 일일 손실 5% → 즉시 중단 / 5th%ile 밴드 20연속 하회 → 리뷰

### D-4. 연구 결과 검증

| 항목 | 검증 방법 | Pass 기준 |
|------|----------|----------|
| **출처 검증** | DOI 해소, 저자 확인, 저널 확인 | 인용 100% 실존 |
| **내용 일치** | 인용된 소스가 실제로 해당 주장을 하는가 | 맥락 왜곡 0건 |
| **CRAAP 점수** | 시의성·관련성·권위·정확성·목적 | ≥35/50 |
| **교차 검증** | 핵심 주장 ≥3개 독립 소스 일치 | 단일 소스 의존 0건 |
| **AI 환각 점검** | 시대착오, 엔티티 혼동, 조작 통계 | 환각 0건 |
| **AMSTAR 2** (SR) | 7개 필수 항목 통과 | 필수 1개라도 실패 → "낮음" |

### D-5. User-Eye 사용자 경험 검증

> "기술적으로 맞아도 사용자가 화낼 수 있다." — harsh-critic 원칙

코드와 시스템이 아닌 **사용자 관점**에서 산출물을 검증한다.
visual-proof Phase 3(Evaluator 채점)과 연동하되, 비-UI 산출물(API, 문서, 설정)에도 적용.

| 항목 | 검증 기준 | Pass 기준 |
|------|----------|----------|
| **거짓 완료 여부** | 실제 브라우저/API/DB 테스트를 수행했는가 | 검증 증거(스크린샷/응답 로그) 존재 |
| **떠넘기기 여부** | "확인해주세요"로 끝나지 않고 직접 검증했는가 | 사용자 액션 요구 0건 |
| **변경 범위 완전성** | 연쇄 수정이 필요한 파일을 전부 수정했는가 | 타입/설정/문서 연쇄 누락 0건 |
| **로딩 피드백** | 비동기 작업 중 사용자에게 진행 표시가 있는가 | 3초+ 대기 시 스피너/스켈레톤 존재 |
| **에러 상태 처리** | 네트워크 오류/빈 데이터 시 사용자에게 안내되는가 | 빈 화면/무반응 0건 |
| **디바이스 일관성** | Desktop/Mobile 모두 핵심 기능이 동작하는가 | 디바이스간 기능 격차 0건 |
| **초기 상태** | 데이터 0건일 때 empty state가 존재하는가 | "고장났나?" 오인 가능성 0건 |
| **반복 실수** | 누적 교훈(ALWAYS/NEVER)에 기록된 문제가 재발했는가 | 재발 0건 |

**채점:** 8개 항목 × Pass(1)/Fail(0) → 100점 만점 환산.
**합격선:** ≥75 (6/8 통과). 거짓 완료/반복 실수는 필수 항목(Fail 시 즉시 No-Go).

## Phase 5: Go/No-Go 게이트 판정

### 판정 매트릭스

```
필수(Must-Meet) 기준: 1개라도 Fail → No-Go (즉시 차단)
선호(Should-Meet) 기준: 1~4점 가중 평균 > 2.5 → Go

판정 결과:
  🟢 Go          — 모든 필수 통과 + 선호 평균 ≥2.5 → 진행
  🔴 No-Go       — 필수 1개+ 실패 → 이전 단계로 복귀 + 구체적 피드백
  🟡 Conditional  — 필수 통과, 선호 일부 미달 → 조건부 진행 (기한 내 시정)
  ⏸️ Hold         — 외부 입력 대기 필요 → 일시 중지
  ⛔ Kill         — 근본적 결함 → 이 경로 포기
```

### 0~100 검증 점수 산출

```
점수 = Σ (각 항목 점수 × 가중치) / Σ 가중치 × 100

등급:
  90~100: ★★★ Excellent — 즉시 진행
  70~89:  ★★  Good — 경미한 시정 후 진행
  50~69:  ★   Fair — 상당한 시정 필요
  <50:    ☆   Poor — No-Go, 재작업 필요
```

## Phase 5.5: 교차검증 에스컬레이션 (v2.5)

L2 이상 리고르에서 도구 간 교차검증을 강제한다.

| 리고르 | 필수 교차검증 | 에스컬레이션 |
|--------|-------------|-------------|
| L0~L1 | 단일 도구 OK | — |
| L2 | g-stack /review + bkit gap-detector | 불일치 시 /codex 추가 |
| L3 | g-stack /review + /cso + bkit + /codex | 불일치 시 unbounded 평가 모드 |
| L4 | L3 전체 + 외부 전문가 리뷰 | 불일치 시 Go/No-Go 보류 |

**교차검증 규칙:**
- 2개 도구가 동일 이슈 지적 → 확신도 HIGH
- 1개 도구만 지적 → 확신도 MEDIUM, 수동 확인
- 도구 간 상반 의견 → 확신도 LOW, 사용자 판단 요청 (User Sovereignty)
- WHEN: sonnet으로 2회 실패 THEN: opus로 모델 에스컬레이션
- WHEN: opus로도 실패 THEN: /codex 교차검증
- WHEN: /codex도 실패 THEN: unbounded-engine 재진입 (문제 재정의)

## Phase 6: 메타 검증 (검증의 검증)

| 차원 | 점검 내용 | 주기 |
|------|----------|------|
| **완전성** | 모든 요구사항에 검증 증거가 있는가 | 매 게이트 |
| **유효성** | 탈출률(프로덕션 결함 ÷ 전체 결함) 추적 | 분기별 |
| **독립성** | 리고르 레벨에 맞는 독립성 확보 여부 | 매 릴리즈 |
| **프로세스 준수** | 검증 절차가 실제로 따라졌는가 | 매 릴리즈 |
| **임계값 교정** | Go/No-Go 예측 vs 실제 결과 추적 → 조정 | 분기별 |

### 메타 검증 규칙
- 모든 게이트 수동 무시(override)는 **2인 승인 + 후속 티켓 생성 + 감사 추적** 필수
- 매 롤백 시 **비난 없는 사후 분석(blameless post-mortem)** 실시
- 보수적으로 시작하여 데이터 축적에 따라 임계값 완화
