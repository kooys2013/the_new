# Install Planner — 코드 변경 사전 분석 스킬

## 절대 원칙

```
1. 추정 금지 — 파일 구조, 변수명, 함수 시그니처를 추정하지 않는다
2. 코드베이스 실측 — 반드시 Read/Grep/Bash 도구로 실제 코드를 읽고 인용한다
3. 매뉴얼 선행 — 코드 수정 전에 매뉴얼을 먼저 작성한다
4. 승인 필수 — 사용자 승인 없이 코드를 수정하지 않는다
5. 아이디어 문서화만 — 아이디어는 문서에 기록만 하고 실행하지 않는다
6. 블랙리스트 보호 — 보호규칙이 있는 구문은 절대 승인 없이 수정하지 않는다
7. GATE 준수 — 수정/변경 시 거짓보고 방지 GATE를 반드시 통과한다
```

### 🔴 절대 금지 행위

```
1. 컨텍스트/요약 정보만으로 코드 구조 판단
2. 가정 기반 추론 ("A가 이러니 B도 그럴 것이다")
3. 일부 폴더만 확인 후 전체 결론 도출
4. 코드 확인 없이 "없음", "미준수" 주장
5. 검색 결과 없음 = 파일 없음으로 단정
6. 주석만 보고 기능 구현 여부 판단
7. 이전 Phase 미완료 상태에서 다음 Phase 진행
8. GATE 미통과 상태에서 "완료" 보고
```

---

## 실행 흐름

```
사용자 요청
    │
    ▼
[Phase 0] 코드베이스 실측 분석 ← Read/Grep으로 실제 코드 읽기
    │
    ▼
[Phase 1~N] 파스별 상세 플랜 작성 ← 코드 스니펫 기반
    │
    ▼
[저장] 매뉴얼 .md 파일로 저장
    파일명: 작업진행/{순번}_{제목}_{YYYYMMDD_HHMM}.md  ← KST 기준
    폴더 없으면 생성 후 저장
    │
    ▼
매뉴얼 제출 + 승인 요청
    │
    ├─ ✅ 승인 → Phase 순서대로 실행
    ├─ 🔧 수정 → 매뉴얼 재작성 + 재저장
    └─ ❌ 취소 → 작업 중단
```

---

## Phase 0: 코드베이스 실측 분석

**이 Phase에서는 코드를 수정하지 않는다. 오직 읽기만 한다.**

### 0-1. 프로젝트 구조 스캔

반드시 아래를 실행하고 결과를 매뉴얼에 포함:

```bash
find src/ -type f -name "*.ts" -o -name "*.tsx" | head -50
cat package.json | grep -A 5 '"dependencies"'
```

### 0-2. 수정 대상 파일 전수 읽기

수정이 필요한 파일을 반드시 Read/Grep으로 읽는다.

**추정 금지 예시:**
```
❌ "candleSeries라는 변수가 있을 것으로 보입니다"
✅ "185줄: const candlestickSeries = { type: 'candlestick', ... }"
```

### 0-3. 타입 호환성 실측

관련 타입을 grep으로 직접 확인한다.

### 0-4. 충돌 검사

```bash
# 동일 함수명 사용 여부
grep -rn "[함수명]" src/
# 동일 import 경로
grep -rn "from.*[모듈명]" src/
```

### 0-5. 블랙리스트 보호 검사

```bash
grep -rn "DO NOT MODIFY\|PROTECTED\|BLACKLIST\|LOCKED" src/
```

## Phase 1~N: 파스별 상세 플랜

**각 Phase는 반드시 아래 4개 섹션을 모두 포함한다.**

### Phase 템플릿

```
### Phase N: [작업 제목]

#### 1. 진행 플랜

대상 파일: [경로]
수정 위치: [줄 번호] [함수명/블록명]

수정 전 (실측 코드):
// [파일경로]:[시작줄]-[끝줄] (실측)
[Read로 읽은 실제 코드]

수정 후:
// [파일경로]:[시작줄]-[끝줄] (수정 후)
[변경된 코드]

#### 2. 위험도 평가

위험도 기준:
🔴 높음 — 기존 기능 깨질 가능성
🟡 중간 — 특정 조건에서 문제 가능
🟢 낮음 — 거의 위험 없음

+----------+-------+------------------+----------+
| 위험 항목 | 등급  | 근거 (파일:줄)   | 대응     |
+----------+-------+------------------+----------+
| [항목]   | 🟡    | [실제 코드 인용]  | [대응]   |
+----------+-------+------------------+----------+

#### 3. 체크리스트

□ 수정 직전에 Read로 대상 파일 다시 읽기
□ [구체적 수정 항목 1]
□ [구체적 수정 항목 2]
□ npx tsc --noEmit 에러 0
□ npm run build 성공

#### 4. E2E 테스트

1단계: npm run build 에러 0 확인
2단계: 기존 기능 회귀 테스트 3개 이상
3단계: 신규 기능 동작 테스트
4단계: 브라우저 콘솔 에러 0개
```

## 매뉴얼 저장 규칙

```
저장 경로 : 작업진행/
파일명 형식: {순번}_{제목}_{날짜시간}.md

순번     : 작업진행/ 폴더 내 기존 .md 파일 최대 순번 + 1 (3자리 제로패딩)
제목     : 수정 내용 요약 (한글 가능, 공백→_, 최대 30자)
날짜시간 : YYYYMMDD_HHMM (KST UTC+9 기준)

예시     : 작업진행/009_P레벨패널구현_20260321_1530.md

폴더 없으면 생성 후 저장.
저장 완료 후 승인 요청 출력.
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

총 Phase: N개
신규 파일: N개 [목록]
수정 파일: N개 [목록]
최고 위험도: 🟡 Phase N — [항목]

1. ✅ 전체 승인 → Phase 1부터 순서대로 진행

3. ❌ 취소 → 작업 중단
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 거짓보고 방지 GATE (6단계)

```
Gate 0: 계획서 요구사항 전수 확인
Gate 1: Grep 검증 — 추가 함수/변수가 코드에 실제 존재
Gate 2: 파일 존재 검증 — 신규 파일 ls로 확인
Gate 3: 실제 코드 확인 — Read로 수정 부분 직접 읽기
Gate 4: Phase 의존성 — 이전 Phase GATE 통과 여부
Gate 5: 검증 보고서 작성 (ASCII 표)

+--------+------------------------+--------+
| Gate   | 검증 내용              | 결과   |
+--------+------------------------+--------+
| 0      | 체크리스트 N/N         | ✅     |
| 1      | grep N/N 확인          | ✅     |
| 2      | 파일 N/N 존재          | ✅     |
| 3      | 코드 대조 N/N 일치     | ✅     |
| 4      | 이전 Phase GATE 통과   | ✅     |
+--------+------------------------+--------+

Gate 6: 전부 ✅인 경우에만 "완료" 보고 가능
```

## 보고서 표 규칙

모든 표는 ASCII 고정폭 표 (code block 안)로 작성한다.

```
✅ 필수:
+----------+--------+--------+
| 항목     | 상태   | 비고   |
+----------+--------+--------+
| A        | ✅     | OK     |
+----------+--------+--------+

❌ 금지:
| 항목 | 상태 | 비고 |
```
